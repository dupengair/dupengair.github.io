三十、是药三分毒——如何用好inline

inline 函数，what a good idea!, 看起来像函数，动作像函数，比宏好得多，又不会招致函数调用的额外开销，实乃一剂提高coding 性能的良药。

元芳，你怎么看？

一、关于inline

  1. 编译器最优化机制常被设计用来浓缩那些“不含函数调用”的代码，所以当inline某个函数，可能编译器有能力对函数的执行语境相关最优化。
    • side effects：
      • 编译器对inline函数的每一个调用都以函数本体替换，这样可能增加目标代码大小。inline造成的代码膨胀亦会导致额外的换页行为（paging），降低指令高速缓存的命中率（instruction cache hit rate），以及相伴而来的效率损失。
    • 正能量：
      • 如果inline函数的本体很小，编译器针对函数本体产生出的代码可能比针对函数调用产生出的代码更小，这样将函数inline确实可能导致较小的目标代码和较高的指令高速缓存命中率。

二、inline的定义

inline只是对编译器的一个申请，不是强制命令，可以隐式提出，也可以显式提出。

  • 隐式提出：
    • 将函数定义于class定义中
    • 当friend函数定义于class中，也会被隐式声明为inline

三、inline与template

inline函数和template通常都被定义于头文件中：

  1. inline函数通常一定定义在头文件中，因为编译过程进行inline时将一个函数调用替换为inline函数的本体，需要编译器知道函数的定义。少数编译环境如.NET CLI的托管环境可以在运行期完成inline，不过在大多数C++程序中inline过程是编译期行为。
  2. template通常也被置于头文件内，在使用时编译器将模板特化，需要知道模板的定义。
  3. template的特化与inline无关，当需要时才将template函数声明为inline

四、编译器不能inline的情况

如上所述，inline只是对编译器的一个申请，是否inline主动权在编译器上面。

  1. 大部分编译器拒绝将过于复杂的的函数inline，如含有循环或递归
  2. virtual函数不会被inline，因为virtual意味“等待，知道运行期才确定调用哪个函数”，而inline意味“执行期，先将调用动作替换为被调用函数”

  3. 如果程序要取得某个inline函数的地址，编译器通常为函数生成一个outlined函数体，因此，编译器通常不为“通过函数指针进行调用”的函数进行inline。

    如：
inline void f() {}  //有意向编译器申请inline的函数
void (*pf)() = f;   //pf指向f
…
f();            //这个调用被inline
pf();            //这个不被inline，因为通过指针调用

  4. 程序员并非唯一要求指针函数的人，有时候编译器会生成构造函数和析构函数的outline副本。由于编译器在编译期间主动生成并添加到程序中的代码有时候就在构造和析构函数中。

五、pay attention to inline

  1. inline 函数无法随程序库的升级而升级，一旦程序库设计者决定改变，所有用到inline函数的客户端程度都要重新编译，如果是outline函数则只需要重新连接。
  2. 大部分调试器对inline函数都素手无策，因为不可能在一个不存在的函数中设置断点。

六、合理使用inline

  1. 一开始先不要将任何函数声明为inline，或至少将inline限定为一定会inline的函数上。
  2. 作为一名软件开发者，目标是找出可以有效增进程序整体效率的20%代码，然后尽力inline或瘦身，但除非选对目标，不然就是徒劳。
