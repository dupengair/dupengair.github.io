一、设计原则

  1. 单一职责原则（SRP）
    • 就一个类而言，应该仅有一个或者一类引起它变化的原因。如果一个类承担的职责过多，就等于把这些职责耦合在一起，一个职责的边和可能会削弱或则抑制这个类完成其它职责的能力。这种耦合会导致脆弱的设计，当变化发生时，设计会收到意想不到的破坏。
    • 软件的设计要做的内容，就是发现职责并把职责相互分离，如果能够有多于一个动机去改变一个类，那么这个类就具有多于一个职责，就应该考虑设计上的职责分离。
    • 例子：界面与业务逻辑分离
  1. 开放-封闭原则
    • 软件实体（类、模块、函数等）应该做到便于扩展，但不可修改，即对于扩展是开放的（Open for extension），对于更改是封闭的（Closed for modification）。在设计上，要求软件面对需求的改变可以保持相对稳定，从而使系统可在在第一个版本后不断演进新的版本。
    • 对于模块设计来说，无论模块多么封闭，都会存在一些无法封闭的变化，设计时必须对于哪种变化封闭作出选择，找到最有可能发生变化的种类，然后构造抽象来隔离这些变化。当变化发生时，通过这些抽象来实现不同的变化，这样对程序的改动是通过增加新代码来进行的，而不是更改现有代码。
    • 开封原则是面向对象设计的核心，遵循这个原则可以使软件可维护、可扩展、可复用、灵活性好。开放时对程序中针对需求表现出变化的部分做出抽象，同时也要注意不要刻意地进行抽象导致过设计，这是新手最易犯的问题之一，拒绝不成熟的抽象和抽象本身一样重要。
    • 例子：计算器、活字印刷
  1. 里氏代换原则（Liskov Principal）
    • 子类型必须能够替换掉它们的父类型。一个软件实体如果使用的是一个父类的话，那么一定适用于其子类，而不会觉察出父类对象和子类对象的区别。也就是说软件里面，把父类都替换成子类，程序的行为没有变化。
    • 只有当子类可以替换父类，软件的行为不受影响是，父类才能真正的被复用，而子类也能够在父类的基础上增加新的行为。而正是由于子类型的可替换性，才使得父类型在不需修改的情况下就可扩展。
    • 这即常说的继承要满足is-a的关系。
    • 例子：访问数据库，动物

  1. 依赖倒置原则（DIP）

    • 高层模块不应依赖低层模板，两个都应依赖抽象
    • B、抽象不应依赖细节，细节应依赖抽象
    • 设计软件时，考虑代码复用，将不同功能的模块分层设计，高层模块通过接口类访问调用低层模块，而不去关心它的实现，参考里氏代换原则。如系统接口与业务逻辑。
    • 高层模块 <—> Interface类 <—> 底层模块
    • 类抽象出的接口只影响调用它的模块行为，不应决定如何去实现，* 换句话说，接口固定而实现可以变化，即虚函数的意义。
    • 因此，当设计一个类时，最重要的是对接口的设计，这决定了依赖它的高层模块的行为，而实现只是对接口设计的完善，在上层应用中，并不依赖于具体的实现。
    • 依赖倒转原则是面向对象设计的标志，与使用哪种语言来实现无关，如果编写时考虑的都是针对抽象编程而不是针对细节，即程序中所有的依赖关系狗屎终止与抽象类或者接口，就是面向对象设计，否则就是过程化设计。

二、设计过程

  1. 寻找合适的对象

    • 对象包括属性和操作，在收到客户请求后执行相应操作。客户请求是使对象执行操作的唯一办法，而操作又是对象改变内部数据的唯一方法，对象的内部状态总是被封装起来，不能直接访问。
    • 面向对象的设计过程就在于将系统分解成对象集合，而抽象通常来自于现实世界的业务逻辑的分析模型。

  2. 决定对象的粒度

    • 将过程抽象成对象的特定方法，合理运用设计原则

  3. 指定对象接口

    • 对象接口描述了对象能接受的全部请求的集合，任何匹配对象接口的请求都可以发送给对象，对象只有通过接口才能与外部通讯。
    • 在面向对象的设计中，对象接口与其功能实现是分离的。当给对象发送请求时，所引起的具体操作既与请求本身有关，又与接受对象有关，支持相同请求的不同对象可能激发相同操作的不同实现。
    • 发送给对象的请求和它的响应操作在运行时刻连接为动态绑定，这使得在运行时刻可以替换有相同接口的对象，这种可替换性即为多态。多态使得对象间相互独立，并且可以在运行时刻改变它们的相互关系。

三、考虑一些问题

  1. 抽象类的目的是为子类定义公共接口。重定义父类的接口可以使子类能接管父类的操作。

  2. 区分接口继承与实现继承

    • 接口继承描述了一个对象什么时候能被用来代替另一个对象，如公有继承一个只含接口的抽象类。
    • 实现继承根据一个类的实现定义了另一个类的实现，如私有继承。
    • 接口的继承的方式：
      • 只继承成员函数的接口（pure virtual）
      • 同时继承接口和实现，又希望能覆盖继承的实现（virtual）
      • 同时继承接口和实现，不允许覆盖任何东西（non-virtual）

  3. 对接口编程，而不是对实现编程

    • 不声明某个具体类的实例对象，而是指向抽象类的指针或引用，让它遵循抽象类定义的接口，当对象实例化时建立接口和实现的透明连接。

  4. 优先使用组合，而不是继承

    • 类继承允许根据根据其它类的实现定义一个类的实现，这种复用为白箱复用（white-box-reuse），父类的内部细节对子类可见。
    • 对象组合要求被组合的对象由良好定义的接口，其内部细节是不可见的，这种复用为黑箱复用（black-box-reuse）。
    • 继承在编译时刻静态定义，子类可以方便的改变被复用的实现，但继承通常会破坏封装性，使父类与子类紧密耦合，父类的任何变化会导致子类发生变化，不利于扩展，即使使用抽象类，也不能完全避免父类的改动。
    • 对象组合是通过获得其它对象的引用而在运行时刻动态定义的，因为对象只能通过接口访问，不会破坏封装性，运行时刻可以替换被组合的对象。因为对象的实现是基于接口的，所以依赖关系较低。
    • 优先使用组合有助于保证每个类被封装并且指责单一，系统将会存在更多的对象，系统行为将依赖于对象将的关系而不是被定义在某个类中。并且不用为了复用去创建新的构建，一个对象可被多个对象复用。

  5. 委托

    • 在处理请求时，接受请求的对象将请求执行的操作委托给代理者来执行，便于在运行时刻组合对象操作，以改变这些操作的组合方式，即代理的思想。

  6. 参数化类型

    • 参数化类型运行改变类所用到的类型，实现方法如下：
      • 通过子类实现；
      • 通过组合对象实现
      • 通过类模板实现

四、设计规范

  1. 分清编写的类

    • 几个特殊用途的类：
      • 值类：用于表达类型和捆绑的操作
        • 公共析构函数、拷贝和复制
        • 无虚函数（包括析构），用作具体类，而不是基类
        • 总在栈中实例化，或者作为成员被包含
      • 基类：用于约定对外接口
        • 公有虚析构，非公拷贝赋值
        • 通过虚函数建立接口
        • 总是动态堆中实例化，通过（智能）指针操作
      • trait类：用于表现确定的类型特性
        • 通常实现为模板，并且禁止实例化（禁止构造），只包含typedef和静态接口，状态不能修改
      • policy类：可插拔的行为片段
        • 通常实现为模板，不独立实例化，只作为基类或者默认模板参数（依赖型）
      • 异常类：
        • 公共析构函数，构造函数和拷贝构造函数总是成功的
        • 最好从std:exception派生

  2. 用小类代替巨类

    • 单一职责原则
    • 注意粒度
    • 易于理解、部署
    • 易于封装、不易出错

  3. 用组合代替继承

    • 继承是C++除friend外能表达的最强关系

  4. 避免继承自不是设计成基类的类

    • 无虚析构、无虚函数、值类

  5. 优先设计抽象接口

    • 依赖倒置原则（DIP）
    • 二次机会定律（Law of Second Chance）：类设计中最重要的是接口，实现如果错了很好修改，而接口一旦提供则无法在不影响客户使用的情况下修改

  6. 继承：不是为了重用，是为了被重用

    • 里氏代换原则（Liskov Principal）：不是使派生类重用基类实现，而是在使用基类的情况下可以重用派生类的实现
    • 经典的正方形长方形例子

  7. 虚函数：实施安全的覆盖行为

    • 重写虚函数时，保持基类的可替换性，即不改写基类行为（里氏代换原则）
    • 派生类重写基类的虚函数，要做到要求更少，而提供更多，而不是要求更多而承诺更少
    • 如：默认参数、异常

  8. 考虑NVI方法或template模式

    • 纯粹的抽象：非虚函数做公共接口，虚函数用来改写实现
    • 需要接口固定时

  9. 避免提供隐式类型转换

    • 防御性编程，如string类c_str()方法

  10. 私有数据成员，合理使用pImpl

    • 使私有成员真正不可见，提高封装性和扩展性
  1. 优先使用非成员非友元接口
    • 提高封装性，减少耦合，避免巨类

12、new和delete成对出现

* 除了使用上的好处，还包括异常处理等场景下

  1. 定义类的专属operator new时，要完全——尤其是准备用于STL容器的类型时

    • 普通、placement、nothrow

      void* operator new(std::size_t);               // Plain
void* operator new(std::size_t, void*);        // placement
void* operator new(std::size_t) throw();    // nothrow

五、设计应解决的问题

  1. 显示指定一个类来创建对象，是设计受特定实现的束缚而不是接口的束缚

    A a;  ->  base* p;

  2. 对特殊操作的依赖

    a.f(); ->  p->f();

  3. 对硬件和软件平台的依赖

    • 分层隔离，依赖倒置
  4. 对对象表示或实现的依赖
    • 委托，针对接口编程
  5. 算法依赖
    • 同上
  6. 紧耦合
    • 抽象分层，组合委托
  7. 通过继承来扩展功能
    • 组合委托，避免类爆炸
  8. 扩展性
    • 原则一，对扩展开放，对修改封闭
