#一、分析服务器性能

  1. 分析方法

    • 创建性能基线
    • 识别性能瓶颈，缩小瓶颈到子系统水平
    • 应用程序编译优化，以减少优化路径
    • 配置硬件最优化
    • 系统性能调优
    • 优化前后的性能对比

  2. 分析对象

    • 处理器
    • 系统
    • 服务器工作队列
    • 存储器
    • 分页文件
    • 物理磁盘
    • 重定向
    • 网络接口

#二、CPU瓶颈
整体查看系统工作状态，高cpu并不意味cpu消耗在实际工作任务

  1. 查找瓶颈

    • uptime：过去15分钟的系统负载
    • top：cpu使用率，问题的最大贡献进程
    • sar&gnuplot：cpu使用的趋势
    • vmstat：系统当前使用率

  2. SMP

    • SMP架构下，由于CPU cache优化，绑定进程到一个CPU上，比在多个CPU之间移动的效率更高
    • 进程在CPU间移动，需要刷新CPU cache
    • 根据CPU亲和力，来绑定进程到CPU

  3. 性能调整

    • 使用ps -ef查看是否有不必要运行的程序，用cron安排在非高峰期运行
    • 使用top识别非关键性CPU密集型程序，renice优先级
    • 在SMP架构中，使用taskset绑定进程到CPU，避免进程在cpu间跳跃导致cache刷新
    • 根据应用程序特性，提高cpu性能，还是扩展cpu个数。如单线程更适合提高cpu性能

#三、内存瓶颈

  1. 通情况下，分页大小为4K或8K，定义在内核头文件include/asm-<arch>/param.h，变量EXEC_PAGESIZE
  2. 将进程分页移出到磁盘的过程为pageout(分页移出)

  3. 查找内存瓶颈

    • 可用内存：了解内存泄漏，free -lto
    • 页错误：通过sar -B，观察 pgpgin/s和pgpgout/s
      • 次要页错误：分页在内存中被找到
      • 主要页错误：分页不在内存中，需要从磁盘提取
    • 文件系统cache：free -lto
    • 进程的内存：pmap

  4. Paging(页面调度)和swapping(交换)

    • Paging是移动个别分页到swap空间，swapping是移动整个进程内存空间到swap
    • swap的原因有2：
      • 进程睡眠
      • 空闲内存分页低于系统最低限度，swap机制释放更多分页，会显著增加磁盘IO，减低服务器性能
    • 如果具有高分页移出率，需要考虑增加内存

  5. 性能调整

    • 调整swap空间，似乎呀bigpages、hugetlb共享内存
    • 调整分页大小
    • 改进活跃内存和非活跃内存的处理
    • 调整分页移出率
    • 限制进程资源使用
    • 限制服务数量
    • 增加内存

#四、磁盘瓶颈
最常见的系统瓶颈，可能被别的问题掩盖

采用单片大容量磁盘，导致用户并发访问量大，影响IO速率并发生磁盘瓶颈

或者一个阵列上有太多逻辑磁盘，增加寻道时间并显著降低性能

  1. 查找磁盘瓶颈

    • 磁盘读写速度低：
      • buffer被填满，延迟所有的写请求
      • 内存不足，如网络请求的可用buffer不足，将导致同步磁盘IO
    • 磁盘和控制器使用率高
    • 网络响应时间长、使用率低，因为大多数网络传输需要等待磁盘IO完成(?)
    • 磁盘IO时间长，导致队列拥塞，CPU使用率低

  2. 磁盘读写方式

    • 随机访问还是顺序访问
      • 随机的工作负载需要多个磁盘扩展，如大型数据库受益于更多的磁盘
      • 顺序的工作负载侧重于磁盘子系统的总线带宽
    • 工作负载的吞吐量
      • 大IO与小IO

  3. 跟踪与监控

    • vmstat 1
      • io中的bi与bo
      • 拥有基线是关键
    • iostat -x 1
      • 电梯算法的改变，会反映到avgrq-sz(平均请求大小)和avgqu-sz(平均队列长度)，通过调整电梯算法降低延迟，avgrq-sz会减小
      • rrqm/s和wrqm/s合并了读和写数量上的影响

  4. 性能调整

    • 对于连续访问工作负载，添加性能更好的磁盘控制器，对于随机访问的工作负载，添加更多的磁盘
    • RAID环境中过添加更多的磁盘驱动器，使得数据展开到多个磁盘，提供读写性能。

#五、网络瓶颈

  1. 指标

    • 发送/接收的数据包：网卡交换的数据量
    • 冲突的数据包：同一个域的多个系统产生冲突，如使用hub
    • 丢弃的数据包：被过滤或者硬件设备性能不一致，对性能有影响
    • 错误：通信线路质量差，导致数据包损坏被重新发送，降低吞吐量
    • 适配器故障：可能产生垃圾包

  2. 性能调整

    • 网卡配置与链路上的路由器、交换器一致，如帧大小等
    • 提供网卡性能，或者添加网卡，绑定成适配器群组
    • 合理规划子网
    • 调整内核性能以及安全相关的参数
