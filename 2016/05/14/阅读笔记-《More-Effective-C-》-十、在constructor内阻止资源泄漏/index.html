#一、资源泄漏
class Image;
class AudioClip;

class BookEntry {
public:
    BookEntry(const string& name);
    ~BookEntry();
private:
    string theName;
    Image *theImage;
    AudioClip *theAudioClip;
};

BookEntry::BookEndtry(const stirng& name):theName(0)
{
    if(name != "")
        theImage = new Image(name);
    if(name != "")
        theAudioClip = new AudioClip(name);
}

~BookEntry::BookEndtry()
{
    delete theImage;
    delete theAudioClip;
}
  1. 由于C++保证delete null指针是安全的，所以destructor中删除指针前不必判断其是否为空
  2. C++只会析构已经构造完成的对象，BookEntry对象只有在其constructor执行完毕才算构造完成
  3. 如果AudioClip执行期间发生异常，BookEntry析构函数并不调用，因为BookEntry对象没有构造完成，而theName可以被析构，因为其对象已经构造完成，并且在栈上。

  4. 即使在这个构造函数外捕捉异常，也不能析构对象，如：

    void testBookEntryClass()
{
    BookEntry *pb = 0;
    try {
        pb = new BookEntry("Name1");
        ……
    }
    catch (...) {
        delete pb;        // none sense!
        throw;
    }

    delete pb;
}
    • 在BookEntry构造函数外捕捉异常并不能解决问题，因为当构造函数发生异常时，BookEntry对象并不存在，pb为空，delete pb的行为不会有任何作用。

#二、解决之道

  1. C++为避免删除未完成对象的开销，需要自己手动处理未完成构造的对象。

  2. 通常只需要将所有可能的exceptions捕捉起来，执行必要的清理后重新传播出去。

    BookEntry::BookEndtry(const stirng& name):theName(0)
{
    try {
        if(name != "")
            theImage = new Image(name);
        if(name != "")
            theAudioClip = new AudioClip(name);
    }
    catch (...) {
        cleanup();        
        throw;
    }        
}

~BookEntry::BookEndtry()
{
    cleanup();
}

BookEntry::cleanup()
{
    delete theImage;
    delete theAudioClip;
}
  1. date member的constructors调用其它可能抛出异常的函数，那么这些constructors就必须负责处理异常，在传播之前做必要的清理工作

#三、处理常量指针
class BookEntry {
public:
…
private:
…
Image const theImage;
AudioClip const theAudioClip;
};

  1. 这样的对象必须通过initialize list初始化 

    BookEntry::BookEndtry(const stirng& name):theName(0)，
        theImage(name != "" ? new Image(name) : 0),
        theAudioClip(name != "" ? new AudioClip(name) : 0)
{
    ...
}

  2. 此时无法用过try/catch语句来解决问题，因为initialize list中要求表达式

  3. 此时只能将try/catch语句安排到其它地方

    BookEntry::BookEndtry(const stirng& name):theName(0)，
        theImage(initImage(name)),theAudioClip(initAudioClip(name)0)
{
    ...
}

Image* BookEntry::initImage(const string& name)
{
    if(name != "")
        return new Image(name);    
    else
        return 0;
}    

AudioClip* BookEntry::initAudioClip(const string& name)
{
    tru {
        if(name != "")
            return new AudioClip(name);    
        else
            return 0;
    }
    catch (...) {
        delete theAudioClip;        
        throw;
    }    

}

  4. 这样做的缺点是，由constructor完成的动作散布在各个函数中，造成维护上的困难

#四、以对象管理资源
class BookEntry {
public:
…
private:
…
cosnt auto_ptr theImage;
const auto_ptr theAudioClip;
}; 

BookEntry::BookEndtry(const stirng& name):theName(0)，
        theImage(name != "" ? new Image(name) : 0),
        theAudioClip(name != "" ? new AudioClip(name) : 0)
{
    ...
}
  1. 如果theAudioClip在初始化期间有任何exception抛出，theImage已经是完整构造好的对象，所以会被自动销毁。这是将使用堆资源的指针转换为栈上的对象的好处。
  2. 如果以智能指针对象来取代pointer class members，可以避免在constructor内部出现exception时资源泄漏，不再需要在destructor内手动释放资源，并允许const member pointers和non-const member pointers有一样的处理方式。
