关于Helper

1. 关于Thread
@ 将线程设置作为单独类与Event并列作为Thread类的工具类
@ Thread类继承Event和线程设置类，使用这两个工具类
@ 是否可携带私有数据
@ 可分析线程性能，各线程的工作情况
@ 具有C++11 的packed_task功能
@ 线程分为static和auto，static在系统启动时分配到线程池中，避免运行时开销，使用pool来管理线程，在线程中增加通讯功能，使一个模块的线程可以直接与另一个模块的线程通讯。
@ 线程取消点和取消状态， 动态改变优先级，线程状态
@ Event与Thread分离的设计十分合适？线程消亡时对应的Event还需要额外管理

关于Framework
@ 可以有module自动安装升级过程，不用被动重启程序,由模块主动寻找Framework来安装，模块要加shell成为独立应用程序
@ 锁的递归与优先级管理
@ Framework Interface 采用NVI方法，方便测试和调试
@ 接口设计：粒度细，测试性代码(instrumentation)非侵入性
@ add hostname to path used by FwLoader in case of more than 1 instances
@ VSContext完全多余
@ log中将记录log格式与使用不同log的具体操作耦合到同一个函数中，没有将逻辑和操作分离
@ CLoader与CLoaderConfig的逻辑可以以pImpl方式实现
@ Log以factory的方式来实现和管理，本地不存储log相关的数据，如ID等
@ 考虑支持静态初始化和动态初始化两种方式：
静态初始化：资源在程序启动时初始化完成，内存占用大，速度快，适合大内存的情况下
动态初始化：运行过程中按使用情况初始化资源，内存按需占用，速度慢（在使用时初始化资源），适合内存有限
@ 需要多个外部程序启动，如sysconfig等，难以管理，受运行环境影响
@ 将Core分为可独立部分，可以针对硬件进行裁剪，增强可移植性
@ 不用的Resource和Module也在初始化时生成，对每个产品应该同时定制ResourceLibrary.xml和ModuleLibrary.xml
@ listeners应该是独立功能逻辑，不应分别嵌入到各个模块（Res. status. element）中
@ 日志可以跟踪特定用户
@ 如何针对不同功能设置权限？怎么实现模块功能的新权限扩充？目前所有类型的权限写死在Users源文件中，需要设计一个权限管理器进行灵活处理

关于module
@ 模块退出时由Framework增加自动垃圾回收操作，避免退出时产生资源丢失影响其它模块运行，如忘记Unlink、removeListener等，这就要求Framework能对模块资源进行管理。
@ 模块实现自己的operator new和delete， 由Framework预分配内存，并且由Framework回收，运行过程中内存大小可控，并且便于管理
@ 现在添加一个module，要基于编译环境手动修改很多文件，可以在编译环境中创建一条自动生成新module的命令，按照固定模板生成一个默认的module框架
@ 模块间数据拷贝存在多路拷贝，根据通讯过程。如果用一个内存池做数据共享会不会更快？
@ 在modprod中抓sighup是否合理？是否应由framework统一捕获signals，通过接口分发到modules？

关于 technic
@ 通过唤醒确定的线程，避免惊群现象(Thundering Herd)
@ 对编译器警告的禁用应该局部化（见C++编程规范第1条， Herb Sutter..）
@ 系统的可配置性
@ 函数参数用 In Out 标识进行区分
@ backtrace 的进化
@ 用宏定义代替防止头文件重复包含的定义
@ 对资源的管理，如锁，内存、线程、文件等，可以仿照ConfigLib中的REGISTER宏将类定义注册进baseclass的方式，这样可以方便管理资源
@ 对于资源，记录它的使用者，拥有者和请求者
@ Exception要统一声明，方便于测试程序
@ 避免iostream的使用，系统api效率更高
@ 规划好异常
@ 高性能的log：使用cookie、buffer，多写内存、少写文件
@ 为单元测试和GDB保留接口（如非inline版本函数）
@ 流转发设计到proxy server进程中，通过共享buffer与主进程交互，避免同一个进程多个工作线程，同时便于集成协议，提高主进程的稳定性

高效的进程间共享内存：
@ 多级buffer，读写分开，互不干扰，提高吞吐量
