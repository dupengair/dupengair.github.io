title: 十一、没有谁比自己更重要——重载赋值运算符千万别忘了自我赋值的情况

“伟大”“光荣”“正确”的GCD教育我们，毫不利己、专门利人的是SB~~~

#一、赋值时可能出现自我赋值的情况
如：
px = py;
a[i] = a[j];

  • 这些自我赋值是别名产生的后果，别名是指有一个以上的方法指向同一个对象。
    • 一般而言如果某段代码操作指向多个相同类型对象的指针或者引用，就要考虑这些对象是否为同一个。
    • 如果两个对象来自同一个继承体系，甚至不需要声明就可以造成别名，因为基类指针可以指向任意派生对象。

#二、证同测试

  • 即在赋值前先判断是否为自我赋值

    如：
class Widget{
    ……
    private:
    Bitmap *pb;
};

Widget& Widget::operator=(const Wdget& rhs)
{
    if (this == &rhs) return;         //如果是自身赋值，就不进行任何操作

    delete pb;                //导致异常安全性的做法
    pb = new Bitmap(*rhs.pb);
    return *this;
}
    • 这段代码判断如果是为自我赋值，就不进行操作，注意下面删除pb对象的操作，会导致异常安全性

#三、异常安全性

  • 考虑异常安全性，如果生成新Bitmap对象的过程中出现异常，那么pb将指向一个空指针，所以这里不能提前删除pb。

    如：
Widget& Widget::operator=(const Wdget& rhs)
{
    Bitmap *pOld = pb;            //先备份原来的pb        

    pb = new Bitmap(*rhs.pb);
    delete pOld；                //赋值成功再删除原来的，在发生异常的情况下，pb还是指向原来的对象
    return *this;
}

#四、提高效率

  • 在后面提到copy and swap技术可以提高效率

    如：
Widget& Widget::operator=(const Wdget& rhs)
{
    Widget temp(rhs);            //备份rhs        

    swap(temp)                //将rhs的副本和this对象中的数据交换
    return *this;
}

#五、总结

  1. 在重载赋值运算符的情况下一定要考虑自我赋值的情况，特别是有引用、指针的操作和对象来自同一个继承体系时。
  2. 解决对象自我赋值的办法包括证同测试和copy and swap技术，证同测试时一定要考虑异常安全性。
