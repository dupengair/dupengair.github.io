cid-call id
did-dialog id
tid-transaction id

#一、tid
这个最简单，每次transaction 更新一个tid，所以每次reg、msg、invite、bye都会update这个值

#二、did、cid

  1. 他们的更新在eXosip_update()函数中[eXosip.c]，由同一个local static变量staic_id进行更新，这个变量每进行一次赋值就会++，cid在did之前赋值，当cid和did<0时表示需要更新，如下：

    if(cid<0)   
cid=++staic_id;
if(did<0)
did=++staic_id;

所以cid总是比did小1，并且cid为odd，did为even。

  1. 从eXosip_update()函数可以看出，只有在did、cid<0的时候才会被更新，即是说，当需要更新这两个值的时候先要让它们=-1

#三、更新cid
根据以上分析，即将cid置为-1的地方会更新cid，只有一个函数：

eXosip_call_init()[Jcall.c]

这个函数被两个函数调用：

eXosip_call_send_initial_invite()[eXcall_api.c]
eXosip_process_new_invite()[udp.c]

即是说，当发送或者接收到一个invite的时候，更新cid，由此可见，cid是由invite产生的

#四、更新did
同理，找出did置-1的地方，有两个函数，在同一个文件中，用法一致：

eXosip_dialog_init_as_uac()[Jdialog.c]
eXosip_dialog_init_as_uas()[Jdialog.c]
  1. eXosip_dialog_init_as_uac()

这种情况是作为Client调用时。
这个函数被如下函数调用：

cb_rcv1xx()[Jcallback.c]
cb_rcv2xx_4invite()[Jcallback.c] <- cb_rcv2xx()[Jcallback.c]
cb_rcv2xx_4subscribe()[Jcallback.c] <- cb_rcv2xx()[Jcallback.c]
eXosip_process_newrequest()[udp.c]

1)、前三个函数是收到1xx，或者invite和subscribe的2xx应答时调用：

(1)cb_rcv1xx():
if ((MSG_IS_RESPONSE_FOR(sip, "INVITE") || MSG_IS_RESPONSE_FOR(sip, "SUBSCRIBE"))
{
……
    if (jd == NULL) {
        eXosip_dialog_init_as_uac();
    }
……
}

这是说，当收到invite或者subscribe的1xx response时，如果还没有建立dialog，就初始化一个，此时会更新did。换句话说为invite的发送者建立一个dialog并更新系统的did。

(2)cb_rcv2xx()

if (MSG_IS_RESPONSE_FOR(sip, "INVITE")) {
        cb_rcv2xx_4invite(tr, sip);
    } else if (MSG_IS_RESPONSE_FOR(sip, "BYE")) {
    ……
    }
    else if (MSG_IS_RESPONSE_FOR(sip, "SUBSCRIBE")) {
        cb_rcv2xx_4subscribe(tr, sip);
    } else if (MSG_IS_RESPONSE_FOR(sip, "NOTIFY") && jn != NULL) {
    ……
    }

在每个cb_rcv2xx_4函数里面也是if (jd == NULL) 时调用，这个过程还是和上面(1)一样，即当收到invite或者subscribe的2xx response时，如果还没有建立dialog，就为invite的发送者建立一个dialog并更新系统的did。因为1xx的应答不是必须的。

2)、eXosip_process_newrequest()的调用过程如下：

eXosip_execute()[eXconf.c] ->
eXosip_read_message()[udp.c] ->
udp_tl_read_message()[udp.c] ->
_eXosip_handle_incoming_message()[udp.c] ->
eXosip_process_newrequest()[udp.c]

在_eXosip_handle_incoming_message()中有如下程序段：

i = osip_find_transaction_and_add_event(eXosip.j_osip, se);
    if (i != 0) {
        /* this event has no transaction, */
        eXosip_lock();
        if (MSG_IS_REQUEST(se->sip))
            eXosip_process_newrequest(se, socket); //初始化新dialog处理request
        else if (MSG_IS_RESPONSE(se->sip))
            eXosip_process_response_out_of_transaction(se);
        eXosip_unlock();
    } else {
        /* handled by oSIP ! */
        return OSIP_SUCCESS;
    }

即如果找不到transaction，当MSG_IS_REQUEST时调用eXosip_process_newrequest()处理newrequest，在处理newrequest中：

if (MSG_IS_NOTIFY(evt->sip)) {
    /* let's try to check if the NOTIFY is related to an existing
       subscribe */
    js = NULL;
    /* first, look for a Dialog in the map of element */
    for (js = eXosip.j_subscribes; js != NULL; js = js->next) {
        if (eXosip_match_notify_for_subscribe(js, evt->sip) == 0) {
            i = eXosip_dialog_init_as_uac(&jd, evt->sip);
            ……
        }
    }
}

当这个request为notify消息时，且是subscribe的notify，才会初始化一个dialog。

  1. eXosip_dialog_init_as_uas()
    这种情况是作为Server调用时。
    这个函数被如下函数调用：

    eXosip_process_new_invite()[udp.c]
eXosip_process_new_subscribe()[udp.c]
_eXosip_insubscription_answer_1xx()[Jresponse.c]

1)eXosip_process_new_invite()
这个可以对应上面的cb_rcv1xx()来看，上面的是说invite的发送者会初始化一个新dialog，这里则是作为Server的接收者收到client的invite消息也会在本地初始化一个新dialog

2)eXosip_process_new_subscribe()
和上面一样，收到一个subscribe消息会在本地初始化一个新dialog。

3)_eXosip_insubscription_answer_1xx()
这个函数被调用的过程如下：

eXosip_insubscription_automatic()[eXinsubscription_api.c] ->
eXosip_insubscription_send_answer()[eXinsubscription_api.c] ->
_eXosip_insubscription_answer_1xx()[Jresponse.c] ->
eXosip_dialog_init_as_uas()[Jdialog.c]

这个是给客户端程序使用的API，从字面上看可能也是处理subscription消息的。

#五、总结

  1. 每次transaction 更新tid，所以每次reg、msg、invite、bye都会update这个值。
  2. cid和did的更新在eXosip_update()[eXosip.c]函数中，由同一个local static变量staic_id进行更新，这个变量每进行一次赋值就会++，cid在did之前赋值，当cid和did<0时表示需要更新。
  3. 当发送或者接收到一个invite的时候，将cid设置为-1，表示期待更新，由此可见，cid是由invite产生的。
  4. 对client来说，当收到invite或者subscribe的1xx response时，为invite的发送者建立一个dialog，将did设置为-1，表示期待更新。
  5. 对client来说，当收到invite或者subscribe的2xx response时，为invite的发送者建立一个dialog，将did设置为-1，表示期待更新。因为1xx的应答不是必须的。
  6. 对server来说，收到client的invite消息会在本地初始化一个新dialog，将did设置为-1，表示期待更新。
  7. 对server来说，收到一个subscribe消息会在本地初始化一个新dialog，将did设置为-1，表示期待更新。
  8. 在主循环中调用eXosip_update()，将3~7中设置为-1的id更新，方法如2中描述。
