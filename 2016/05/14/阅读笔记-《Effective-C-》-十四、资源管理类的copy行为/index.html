title: 十四、资源管理类的copy行为
date: 2016-05-13 17:02:04
categories: [阅读笔记, 《Effective C++》]
tag: 《Effective C++》
toc: true

在任何时候，都要注意类的copy行为，对于资源管理来说就更是如此了。

#一、对非heap-based资源的管理

  1. 对于heap-based资源来说，我们可以用auto_ptr和shared_ptr来管理，但并非所有资源都是heap-based，如在Edge OS中对锁的管理：

    #define _lock(x) { Framework::Helper::Synch::AutoMutex AMtx(&x);
#define _endlock }
// ----------------
// Auto Mutex class
// ----------------
class AutoMutex
{
    private:
        Mutex* Mtx;
        bool ToLock;

    public:
        AutoMutex(Mutex* mtx, bool to_lock=true) 
                : Mtx(mtx), ToLock(to_lock)
        {
            if (ToLock)
                Mtx->Lock();
        }

        ~AutoMutex()
        {
            if (ToLock)
                Mtx->Unlock();
        }
};

  2. 这样定义的目的在于，在进入需要锁的作用域时定义一个AutoMutex对象，将传入的锁上锁，在推出作用域时对象自动析构时解锁。

  3. 这是一个很普遍的做法，对锁使用对象的方法来进行资源管理，但忽略了一个问题，这个RAII对象是可以被赋值的，此时会发生什么情况呢？如：

    AutoMutex Atm1(&mtx);    //将mtx上锁
AutoMutex Atm2(Atm1);    //复制锁资源管理对象

#二、对可复制RAII对象的处理

  1. 禁止复制

    • 当我们不希望一个RAII对象被复制时，我们可以采用条款6的做法，继承一个uncopyable类，比如对于大多数情况，锁资源的管理对象是没必要进行复制的，程序可以改成如下：

      class Uncopyable {            //见条款6
protected:
Uncopyable();
~Uncopyable();

private:
Uncopyable(const Uncopyable&);
Uncopyable& operator=(const Uncopyable&); 
};

      class AutoMutex : private Uncopyable {

      ……
};

      这样就不能复制这个AutoMutex对象了。

  2. 对复制操作祭出“引用计数”大法

    • 如果一个资源管理对象确实想要被复制，可以通过shared_ptr来实现reference-counting copying行为。但要注意，shared_ptr的默认行为当引用计数为0时析构所指对象，所以这里要根据资源管理的方式自定义删除器——一个函数或者函数对象（function object）。

      如：
// ----------------
// Auto Mutex class
// ----------------
class AutoMutex
{
    private:
        boost::shared_ptr<Mutex> MtxPtr;    //用shared_ptr使mutex可复制
        bool ToLock;
    private:
        void ExitUnlock(void)    
        {
            if (ToLock)
            Mtx->Unlock();
        }

    public:
        AutoMutex(Mutex* mtx, bool to_lock=true) 
            : MtxPtr(mtx, ExitUnlock), ToLock(to_lock)    //自定义删除器
        {
            if (ToLock)
            Mtx->Lock();
        }
};

    • 注意这里可以不用定义析构函数，因为默认析构函数会调用每个成员对象的析构函数，使shared_ptr定义的删除器被调用。

#三、复制底部资源

  1. 复制资源管理对象时，进行的是“深拷贝”，要注意完整复制。
  2. copy函数（copy构造函数和赋值运算符）有可能被编译器自动创建，因此除非编译器做的是对的，否则要自己完成。

#四、转让底部资源所有权

  1. 有些时候需要保证只有一个RAII对象指向一个raw resource，可以考虑auto_ptr。
  2. 当这个RAII对象被复制时，资源的所有权从被复制对象转移到目标对象中。
