title: 十五、还原真实的自我——资源管理类中提供对原始资源的访问
date: 2016-05-13 17:02:04
categories: [阅读笔记, 《Effective C++》]
tag: 《Effective C++》
toc: true

#一、何时需要访问原始资源

如：
boost::shared_ptr<Invest> pInv(creatInvest());        //工厂函数返回Invest对象
int capital(const Invest *p);        //需要Invest对象的指针做输入
int fund=capital(pInv);                //error！传入的是shared_ptr<Invest>对象，不是指针！

  • 这时候就需要一个指向原始对象的指针做输入参数，来直接访问原始资源。幸好shared_ptr和auto_ptr都提供get方法显式返回内部的原始指针。如：

    int fund=capital(pInv.get());        //通过get方法显式转换

  • 它们也都重载了指针取值操作符->和*，可以隐式转换至底部原始指针。如：

    string investors=pInv->investor();
string investors=(*pInv).investor();

#二、提供对原始资源的访问方式

  1. 现在看另一个资源管理类，如果这个设备管理的RAII class：

    class Device{
public:
    explicit Device(Handle fd):fd(fd) {}
    ~Device() { close(fd); }        //析构时关闭设备句柄
private:
    Handle fd;
};

void OpenDev(Handle fd);            //打开设备的API函数

  2. 提供一个显式转换函数

    class Device{
public:
    ……
    Handle get() { return fd; }
    ……
};

Handle fd;
Device dev(fd);
OpenDev(dev.get());        //显示返回一个Handle

  3. 提供隐式转换

    class Device{
public:
    ……
    operator Handle() { return fd; }
    ……
};    

OpenDev(dev);            //dev对象被隐式转换为Handle类型

#三、两种方式的比较

  1. 隐式转换更方便，但增加出错机会

    如：
Handle fd2; 
fd2 = dev;        //不是拷贝对Device象，而是隐式转换
    • 如果编写上面语句时不知道fd2的类型，很容易误认为在复制device对象，实际上由于发生了隐式转换，得到的是Handle类型。

  2. 优先使用显示转换

    • 设计接口时应该遵循“让接口容易被正确使用，不易被误用”，显示转换虽然麻烦，但避免了“无意的类型转换”，应该优先考虑

#四、理解
RAII class提供对原始资源的访问并不是破坏其封装性，因为RAII 类并不是为了封装资源而存在，而是为了确保资源释放的正确性。它设计的目的在于隐藏客户不需要的部分，提供必要的方法和数据。
