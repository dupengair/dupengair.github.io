二十六、变量定义的讲究——勿以善小而不为

May 14, 2016 | 阅读笔记《Effective C++》 | Hits
文章目录

变量定义这么简单的事情，在C++里也有讲究。首先要明白，只要定义了一个变量，就要承受它的构造成本，程序离开作用域时承受析构成本。我们要做的就是，尽量延后变量定义的出现时间。

#一、首先看一个过早定义变量的实例

std::string encryptPassword(const std::string& password)
{
    using namespace std;
    string encryptd;                //过早定义变量
    if (password.length() < MinimumPWLength) {
        throw logic_error("password too short");
    }
    ……
    return encrypted;
}
  • 如果执行过程中有异常被抛出，那么定义的变量encryptd就不会被用到，浪费了这个变量定义。

#二、改进版

std::string encryptPassword(const std::string& password)
{        
    using namespace std;
    if (password.length() < MinimumPWLength) {
        throw logic_error("password too short");
    }
    string encryptd;                //在实际用到之前定义变量
    ……
    return encrypted;
}
  • 貌似这样的定义似乎合情合理了，但这里encryptd虽然定义却没有任何初值，于是会先调用default构造函数初始化这个对象，然后在后面通过赋值动作完成对象的赋值。很明显，这样做的效率算不上高，我们应该跳过这个无意义的default赋值过程。

#三、完美版

std::string encryptPassword(const std::string& password)
{
    if (password.length() < MinimumPWLength) {
        throw logic_error("password too short");
    }
    ……
    std::string encryptd(password);            //在定义的同时初始化变量        
    encrypt(encryptd);
    return encrypted;
}
  • 从这里可以看出，延后变量的定义，不应该只到使用变量前，而应该延迟定义直到给它初始化实参为止。不仅要避免构造和析构非必要对象，还可以避免无意义的default构造行为。并且初始化赋初值的过程可以附带说明变量的目的。

#四、关于循环

//方法A：定义在循环外
Widget w；                //一个构造
for (int i=0; i<n; i++) {
    w = 取决于i的某个值;        //n个赋值
    ……
    }                //一个析构            

//方法B：定义在循环内
for (int i=0; i<n; i++) {
    Widget w(取决于i的某个值);    //n个构造 + n个析构
    ……
    }
  1. 由此可以看出，这里其实是n个赋值与n个构造 + n个析构效率的比较。
  2. 除非：赋值成本 < (构造 + 析构成本)，效率高度敏感，否则采用B，因为A会扩大变量的作用域。
