title: 九、virtual是把双刃剑——继承关系中构造和析构的探索
date: 2016-05-13 17:02:02
categories: [阅读笔记, 《Effective C++》]
tag: 《Effective C++》
toc: true

virtual是实现多态的基石，通过滞后联编技术，我们获得了面向对象里威力无穷的多态特征

but the more powerful, the more dangerous.

有时这也像一把达摩克利斯之剑，它时刻提醒着我们，要深入理解C++语言的根基，才能合理运用，所向披靡。否则，你要逆天，必然被反噬。

#一、绝不在构造和析构间调用（派生类的）virtual函数

如下代码，会发生什么事？
class A {
    public:
        A() { test_vt(); }            //基类构造函数中调用virtual函数
        virtual void test_vt() = 0;
    };
class B: public A {
    public:
        virtual void test_vt() { …… };    //派生类中实现这个virtual函数
    };

B b;        //当基类构造函数被调用时，会发生什么事？连接期错误！
  • 当编译时会遇到一个连接期错误，提示“undefined reference to A::test_vt()”，也就是说，B类对象构造函数调用的test_vt是声明于基类A中的，而不是像多态那样调用B中定义的虚函数。而A是个抽象类，test_vt()在A类中为纯虚函数，当然没有定义了。看起来，这时对象b通过基类构造函数并没有像多态的方式去调用派生类B中实现的虚函数，这是为什么呢？

#二、继承关系中构造和析构的探索

  1. 当两个类之间有继承关系时，派生类对象的构造期间，先调用基类构造函数构造其中的基类成分，这时由于派生类对象的成分并没有完成构造，所以此时基类中的virtual函数无法访问派生类中的virtual函数，只能调用基类自己实现的版本，即此时的virtual函数不是virtual函数。
  2. 换句话说，在派生类对象的基类成分构造期间，对象类型是base class，而不是derived class，不止virtual函数会被编译器解析为base class，包括RTTI的dynamic_cast和type_id，也会把调用对象识别为base class。
  3. 所以，对象在派生类构造函数开始执行前，即派生类对象中的基类成分构造完成以前，不会成为一个真正的派生类对象。
  4. 对于析构函数也一样，派生类对象的析构顺序是先派生类再基类，当派生类对象的derived class成分析构完成后，这个派生类对象就成为一个基类对象，只能执行基类中定义的virtual、RTTI版本。

#三、派生类初始化基类成分的小技巧——通过static 成员函数传递信息

如：    
class A {
    public:
        A(const int& params) ;            
        void test_vt() ;            //non-virtual函数
    };
class B: public A {
    public:
    B(int params):A(creatParams(params));    //在初始化基类时通过成员函数传递信息
        ……
    private:
        static int createParams(int Params);    //static 成员函数
    };
  1. 初始化基类时通过成员函数传递信息，比通过初始化列表直接传递参数给基类更可读、更方便
  2. 更重要的是，static成员函数不存在对对象的访问，也就不可能调用对象内还未完成初始化构造的数据成员，保证安全性。
