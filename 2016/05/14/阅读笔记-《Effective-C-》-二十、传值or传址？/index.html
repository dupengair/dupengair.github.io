title: 二十、传值or传址？
date: 2016-05-13 17:02:03
categories: [阅读笔记, 《Effective C++》]
tag: 《Effective C++》
toc: true

这个命题揭示了一条C++程序设计的普遍规律：Prefer pass-by-reference-to-const to pass-by-value，即传址优先。

#一、效率问题

  1. 传值产生的副本由对象的copy构造函数产生，函数返回时副本被析构掉，这个代价是巨大的，特别是有深层次继承关系和组合其它类的时候，所谓牵一发而动全身。

  2. 传址的效率高很多，没有任何构造或析构函数被调用，因为没有任何新对象被创建。注意参数声明中的const非常重要，如：

    boot ValidateHost(const Host& host);

#二、对象切割问题（slicing）

  1. 当一个派生类对象以传值的方式传递给函数参数，而参数类型是基类对象时，只有基类的copy构造函数会被调用，造成这个派生类对象中只有基类的成分被构造，从而使派生类的特化性质被切割，仅仅留下一个基类对象的成分，因为正是基类构造函数构造的。

    如：
//类定义：
class Window {
public:
    …
    std::string name() const;
    virtual void display() const;
};

class WindowWithScrollBars: public Window {
public:
    …
    virtual void display() const;
};

//函数定义：pass-by-value
void DisplayWindow(Window w)
{
    std::cout << w.name();
    w.display();
}    

//函数调用：
WindowWithScrollBars w1;
DisplayWindow(w1);            // slicing!
    • 此时DisplayWindow()中调用的是基类方法Window::Display()，而不是派生类的方法WindowWithScrollBars::Display()，也就是说，由于对象切割问题，pass-by-value的参数传递形式无法实现多态的效果，因为派生类的成分没有传递到函数参数中。

#三、内部类型

  • 对于编译器的内部实现来说，reference往往以指针实现，因此pass-by-reference通常意味的传递的是指针。因此如果对象为内部类型，如int时，pass-by-value往往比pass-by-reference效率高。同时对于STL的迭代器和函数对象来说，习惯上也被设计为pass-by-value。

#四、对象大小

  • 对象小并不意味着其copy构造函数就不昂贵。许多对象——包括大多数STL容器——内含的东西只比一个指针多一些，但复制对象的时候需要承担“复制指针所指的每一样东西”，代价是昂贵的。

#五、效率上的争议——内置类型与用户自定义类型

  1. 即使小型对象拥有并不昂贵的copy构造函数，并不意味着传值的效率就高。
  2. 某些编译器对待内置类型和用户自定义类型的态度截然不同，即使两者的底层表述（underlying representation）相同。
    • 如某些编译器拒绝把只由一个double类型组成的对象放进缓存中，但对内置类型double就没问题，而对于pass-by-reference来说，传递的是指针——内置类型，当然没有问题。

#六、计划赶不上变化——对将来的考虑

  • 对于一个用户自定义类型来说，大小很容易发生变化。一个type目前虽然小，将来可能会变大，因为其内部实现可能改变，甚至当改用另一个C++编译器都可能改变type的大小。所以对于自定义对象来说，pass-by-reference总是靠谱的。

#七、总结

  • 一般而言，可以假设“pass-by-value是合理的”唯一情况就是内置类型、STL迭代器和函数对象，除此之外，pass-by-reference总是靠谱的。
