四十三、处理模板基类中的名称

May 14, 2016 | 阅读笔记《Effective C++》 | Hits
文章目录
  1. 1. 一、一个被继承的基类模板
  2. 2. 二、导致问题的特化
  3. 3. 三、说明
  4. 4. 四、解决问题
  5. 5. 五、深入

C++对基类模板中的名称有特殊处理，先看例子:

一、一个被继承的基类模板

  1. 提供具体操作的普通类型:

    class CompanyA {
public:
    void sendCleartext(const std::string& msg);
    void sendEncrypted(const std::string& msg);
};

class CompanyB {
public:
    void sendCleartext(const std::string& msg);
    void sendEncrypted(const std::string& msg);
};

  2. 使用这些普通类型的基类模板:

    template<typename Company>
class MsgSender {
public:
    void sendClear()        //发送cleartext的统一接口
    {
        std::string msg;
        Company c;
        c.sendCleartext(msg);
    }
    void sendSecret()        // 类似
    {
        ……
    }
};

  3. 增加log功能的派生类模板:

    template<typename Company>
class LogMsgSender : public MsgSender<Company> {
public:
    void sendClearMsg()        //在send操作前增加log操作
    {
        log 操作;
        sendClear();
    }
    void sendSecretMsg()        // 类似
    {
        ……
    }
};
    • 可惜，上面这个派生类模板无法通过编译，原因在于编译器并不知道继承的基类模板会被实例化或者特化成什么样子，所以不会去基类模板中读取名称，即无法访问在基类模板中继承来的sendClear。

二、导致问题的特化

  1. 基类模板可以被特化成任何形式。如，现在有个普通类型没有sendCleartext操作:

    class Company {
public:
    void sendEncrypted(const std::string& msg);
};

基类有个为它特化的版本：
template<>
class MsgSender<Company> {
public:
    void sendSecret()        // 这个全特化的MsgSender版本去掉了sendClear操作
    {
        ……
    }
};

  2. 模板全特化指对于已经确定了的模板参数，再没有其它模板参数可供特化。

    • 对应于这种特化情况，派生类模板是无法处理的，所以无法通过编译。

三、说明

  • 派生类模板无法直接访问基类模板中的名称，是因为C++编译器知道基类模板有可能被特化，有可能特化版本无法提供和一般性模板相同的接口，因此派生类模板拒绝直接访问继承自基类模板中的名称。这是Object Oriented C++走进Template C++的不同。

四、解决问题

解决这个问题，有三个办法:

  1. 在派生类模板对base class函数调用前加上”this->”:

    template<typename Company>
class LogMsgSender : public MsgSender<Company> {
public:
    void sendClearMsg()
    {
        log 操作;
        this->sendClear();        //加上this->，告诉编译器sendClear被继承
    }
};

  2. 使用using声明式

    template<typename Company>
class LogMsgSender : public MsgSender<Company> {
public:
    using MsgSender<Company>::sendClear;        //告诉编译器base class中的sendClear可见
    void sendClearMsg()
    {
        log 操作;
        sendClear();        
    }
};

  3. 明确指出被调用的函数位于base class中

    template<typename Company>
class LogMsgSender : public MsgSender<Company> {
public:
    void sendClearMsg()
    {
        log 操作;
        MsgSender<Company>::sendClear();    //明确告诉编译器sendClear在base class中
    }
};
    • 第三个方法不做推荐，因为强制限定了方法在基类中，如果被调用的是virtual函数，将失去virtual绑定的行为。

    • 从名称可见的角度看，每一种做法的目的都一致——告诉编译器基类模板的任何特化版本都支持一般泛化版本的接口。但如果实际定义了不支持相应接口的特化，会导致无法通过编译:

      LogMsgSender<CompanyZ> zMsgSender;
zMsgSender.sendClearMsg();                //error!

五、深入

实质上，对于基类模板中可能存在的无效名称，编译器的发现时间可能在早期，即解析派生类模板的定义时，排除一切可能发生无效名称的情况；也可能发生在晚期，当那些模板被实例化时。C++的处理方式是尽早发现，做法即是避免从继承的基类模板中寻找名称。
