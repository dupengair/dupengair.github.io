二十二、以操作符符合形式（+=）取代独身形式（+）

May 14, 2016 | 阅读笔记《More Effective C++》 | Hits
文章目录

#一、实现

  1. 要确保操作符的复合形式和其独身形式之间的自然关系存在，较好的处理时以复合形式为基础实现其独身形式。

    class Rational {
public:
    ...        
    Rational& operator+=(const Rational& rhs);
    Rational& operator-=(const Rational& rhs);
};

独身形式的实现：
const Rational operator+(const Rational& rhs, const Rational& lhs)
{
    return Rational(lhs) += rhs;    // copy lhs 产生临时对象再加上rhs
}

const Rational operator-(const Rational& rhs, const Rational& lhs)
{
    return Rational(lhs) -= rhs;        // copy lhs 产生临时对象再减去rhs
}

  2. 这样就只需要维护操作符的复合形式

  3. 如果复合形式是在class的public接口内，那么就不需要以独身形式成为class的friends

#二、一般化

  1. 如果不介意把所有独身形式操作符放在全局范围内，可以利用templates，完全消除独身形式操作符的编写必要

    template<class T>
const T operator+(const T& lhs, const T& rhs)
{
    return T(lhs) += rhs;        // copy lhs 产生临时对象再加上rhs
}

template<class T>
const T operator-(const T& lhs, const T& rhs)
{
    return T(lhs) -= rhs;        // copy lhs 产生临时对象再减去rhs
}

#三、效率问题

  1. 一般而言，复合操作符比对应的独身版本效率高，因为独身版本必须返回一个新对象，程序必须负担这个临时对象的构造和析构成本。而复合版本则是直接将结果写入左端自变量，所以不需要产生衣蛾临时对象来放置返回值。

  2. 如果同时提供某个操作符的复合形式和独身形式，使客户可以在效率与便利性之间做出选择

    Rational a, b, c, d, result;

result = a + b + c + d;        //每次+操作都会产生一个临时对象

result = a;                    //不会有临时对象产生
result += b;                //不会有临时对象产生
result += c;                //不会有临时对象产生
result += d;                //不会有临时对象产生
    • 前者易编程，可读性好，后者效率高。当客户做出选择时，可以确保其操作的语法保存不变。

#四、临时对象与命名对象

  1. 、返回值优化

    比较：
template<class T>
const T operator+(const T& rhs, const T& lhs)
{
    return T(lhs) += rhs;                // 临时对象
}    

和
template<class T>
const T operator+(const T& rhs, const T& lhs)
{
    T result(lhs)；                        // 命名对象
    return result += rhs;
}
    • 第二个template内含了一个命名对象result，意味着返回值优化可能无法实施，而第一个总是适用。

  2. 匿名对象与命名对象

    • 第一个template可能要在函数内消耗一个临时对象，然而，自古以来匿名对象比命名对象更容易消除，所以当面对临时对象和命名对象的选择时，最好选择临时对象。它绝不会比其命名版本消耗更多成本。

#五、总结
操作符的复合版本比其对应的独身版本有更高效率的倾向。身为程序设计者，应两者都提供，作为软件开发人员，如果性能是主要因素，则应考虑复合版本取代独身版本
