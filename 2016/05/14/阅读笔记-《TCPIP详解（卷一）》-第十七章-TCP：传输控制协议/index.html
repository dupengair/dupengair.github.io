#一、服务

  1. TCP提供面向连接的，可靠的端到端字节流服务
    • 面向连接：通信双方彼此交换数据前，必须建立一个TCP连接
    • 字节流服务：通信双方交换8bit字节构成的字节流，TCP不在字节流中插入记录和标识符，也不对字节流内容做任何解释
    • 端到端通信：一个TCP连接只能是端到端的双方通信
  1. TCP的可靠性
    • 数据传递给IP时被分割为适合发送的块，称为报文段
    • 超时重传，报文发出一个段会启动定时器
    • 接收端需要确认，通常会推迟几分之一秒
    • 端到端的首部+数据校验和，有差错时丢弃报文并且不确认收到，等待发送端的超时重传
    • 接收端排序，因为IP数据报可能乱序
    • 接收端排除重复数据，因为IP数据报可能重复
    • 进行流量控制，接收端只运行发送端发送接收缓冲区所能接纳的数据

#二、TCP首部

  1. 不计任选字段的情况下，TCP首部为20B

    | IP首部 20B | TCP首部 20B | TCP数据 |

  2. 报文格式

    | 源端口 16b  | 目的端口 16b  |
|              序号 32b          |
|          确认序号 32b         |
| 首部长度 4b | 保留 6b | URG | ACK | PSH | RST | SYN | FIN | 窗口大小 16b |
| 校验和 16b  | 紧急指针 16b  |
|           选项             |
|           数据             |
    • 源端口、IP，目的端口、IP四元组唯一确定一个TCP连结
    • 序号标识发送的字节流，32bit无符号，到最大值后从0开始
    • 新建连接时，SYN标志变1，序号为主机选择的连结的初始序号ISN(Initial Sequence Number)，要发送数据的第一个字节序号为ISN+1，因为SYN标志会消耗一个序号
    • 确认序号为上次已成功收到的数据字节+1，只有ACK为1时确认序号字段才有效（连结建立后，ACK标志总是被置1）
    • TCP提供全双工服务，所以每个传输方向上的序号都要保持
    • TCP可以表述为一个没有选择确认或者否认的滑动窗口协议。缺少选择是因为TCP首部中的确认序号表示发已经成功收到字节，但不包含确认序号所在的字节
    • 首部长度为首部的32bit字的数量，4bit决定了TCP首部最多60B，正常情况下为20B，有选项字段时长度是可变的

    • flags

      URG：紧急指针有效

      ACK：确认序号有效

      PSH：接收方尽快将报文交给应用层

      RST：重建连接

      SYN：同步序号用来发起连接

      FIN：发送端完成发送

    • 窗口大小用于TCP流量控制，为序号段指明的字节数，这个字节数是接收端期望接收的字节大小，最多65535B

    • 校验和覆盖整个TCP报文段，包括首部和数据，由发送端计算和存储，接收端验证
    • 紧急指针是一个正偏移量，和序号字段的值相加表示紧急数据最后一个字节的序号，用于发送端向另一端发送紧急数据
    • 最常见的可选字段是最长报文大小MSS（Maximum Segment Size），通常在通信的第一个报文段——设置SYN标志的报文段指明本端所能接收的最大长度的报文段
    • 数据部分是可选的：在建立和终止连接时，双方交换的报文段仅有TCP首部。在一方没有数据要发送时，也使用没有任何数据的首部来确认接收。在处理超时的很多情况下，报文段也不带数据。
