#一、模板与泛型（Templates & Generic）

  1. C++ templates最初发展动机是为了建立“类型安全”的容器。
  2. 泛型编程写出的代码和其处理对象彼此独立，如STL的算法。
  3. C++模板机制自身是一套完整的图灵机——可以被用来计算任何可以计算的值
  4. 模板元编程——“在C++编译器内执行并于编译时完成停止执行”

#二、显式接口（explicit）和运行期多态（runtime polymorphism）

  1. 面向对象编程总是以显式接口和运行期多态解决问题：

    class Widget {
public:
    Widget();
    virtual ~Wiget();

    virtual std::size_t size() const;
    virtual void normalize();
    void swap(Widget& other);        //条款25
    ……
};

//客户端程序
void doProcessing(Widget& w)
{
    if (w.size() > 10 && w != someNastyWidget) {
        Widget temp(w);
        temp.normalize();
        temp.swap(w);
    }
}

  2. 显式接口（explicit）在源码中明确可见，客户可以在源码中找出，如参数w支持的Widget接口。

  3. 运行期多态（runtime polymorphism）于运行期根据参数w的动态类型决定究竟调用哪一个函数。

#三、Templates与泛型的世界

  • Templates和泛型与面向对象有根本上的不同，显示接口和运行期多态仍然存在，但重要性降低；隐式接口（implicit）和编译期多态（compile-time polymorphism）更为重要。

#四、隐式接口（implicit）和编译期多态（compile-time polymorphism）
将上面的客户端程序转成函数模板：

template<typename T>
void doProcessing(T& w)
{
    if (w.size() > 10 && w != someNastyWidget) {
        Widget temp(w);
        temp.normalize();
        temp.swap(w);
    }
}

  1. 隐式接口（implicit）是一组有效表达式，表达式自身看起来可能很复杂，但它们要求的约束条件一般直接又明确。如：

    • 对if (w.size() > 10 && w != someNastyWidget)来说，if里面的条件式必须是个bool表达式，或者与之兼容。这意味着：“>”必须能被一个T的size成员返回函数类型和一个int类型调用，同样，“!=”必须能被一个T类型和一个someNastyWidget类型调用，或者是这些参数可以隐式转换的值所调用。并且，如果&&被重载，必须保证能接受“>”和“!=”返回结果。
    • T类型必须有copy constructor、normalize和swap接口必须对T类型有效。

  2. 编译期多态（compile-time polymorphism）表现出“以不同的template参数实例化函数模板，会生成和调用不同的函数”。如凡是涉及参数w的任何函数调用都可能导致函数模板被实例化，从而被成功调用，这样的行为发生在编译期。

#五、差异

  1. 显式接口通常由函数的签名式（声明）构成，隐式接口通常由有效表达式组成。template参数带来的隐式接口和class对象定义的显式接口都在编译期完成检查。
  2. 运行期多态和编译期多态的差异类似于“哪一个重载函数被调用”或“以不同的类型实例化模板”（编译期多态）和“哪一个virtual函数被绑定”（运行期多态）

#六、总结

  1. class和template都支持接口和多态（polymorphism）
  2. 对class而言，接口是显示的（explicit），以函数签名为中心。多态通过virtual函数发生于运行期。
  3. 对template参数而言，接口是隐式的（implicit），基于有效表达式。多态通过模板实例化和函数重载解析（function overloading resolution）发生于编译期
