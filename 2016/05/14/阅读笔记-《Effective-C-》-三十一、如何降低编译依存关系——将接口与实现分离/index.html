三十一、如何降低编译依存关系——将接口与实现分离

May 14, 2016 | 阅读笔记《Effective C++》 | Hits
文章目录

#一、警示：糟糕的编译依存关系

#include <string>
#include "date.h"
#include "address.h"


class Person {
public:
    Person(…);
    …
    std::string name() const;
    std::string birthDate() const;
    std::string address() const;
private:
    std::string theName;    //warning! dependency
    Date theBirthDate;        //warning! dependency
    Address theAddress;        //warning! dependency
};
  • 可以看到，在这个类定义中存在严重的编译依赖关系，即使仅仅是程序实现的private部分有所变化，都会造成所有包含类定义的客户端程序重新编译。原因有两方面：
    1. 在private部分包含了太多class的实现细节，当class发生变化时，最容易受到改动的就是private部分的实现，在类定义中的private部分极有可能产生改动，从而影响到所有包含这个类定义的客户端代码都重新编译。（造成外部代码对类的依赖）
    2. 在成员变量定义中直接定义了对象，编译器解析类定义时，会寻找这些成员变量的对象的类定义，造成对这些外部类定义的依赖。因为对于对象的定义来说，编译器在编译期间必须知道对象的大小，编译器获得对象大小的唯一办法就是访问类定义（造成类对外部代码的依赖）

#二、pimpl idiom方法

  1. pimpl idiom——pointer to implementation，在main class内只含一个指针成员指向其实现类，这种class内的指针名往往就是pImpl。

    #include <string>
class PersionImpl;
class Date;            //初始化Person
class Address;            //初始化Person

class Person {
public:
    Person(…);
    …
    std::string name() const;
    std::string birthDate() const;
    std::string address() const;
private:
    std::tr1::shared_ptr<PersonImpl> pImpl;        //pointer to implementation class
};

  2. 分离的关键在于以“声明的依存性”替换“定义的依存性”，这样的设计的好处多多：

    • class的任何实现修改都不需要包含Person的客户端重新编译，将编译依存关系降到最低。
    • 客户无法看到class的实现细节，就不会写出和这些细节相关的代码，真正的将“接口与实现分类”。

#三、降低编译依存关系

  1. 编译依存关系最小化的本质：让头文件尽可能自我满足，否则至少让头文件与其它文件内的声明相依，而非定义。

  2. 使用对象的reference或者pointer，而非定义对象本身。可以只用一个类的声明就定义出对象的reference或者pointer，而如果要定义对象本身，就需要用到对象的类定义。

  3. 尽量以类声明替换类定义。当声明一个函数需要用到某个类时，并不需要这个类的定义。

    class Date;
Date today();
void findDate(Date d);
    • 即如果能将类定义的头文件包含从声明函数的地方转移到客户端调用函数的地方，就可以把不需要的类定义从函数声明中转移到客户端代码。

  4. 为类声明和定义提供不同的头文件

    上面的代码可变为：
#include "datefwd.h"
Date today();
void findDate(Date d);
    • 只含声明的头文件命名“xxxfwd.h”取自C++ STL库文件，例如<iofwd>包含<sstream>、<streambuf>、<fstream>、<iostream>，本条款同样适用于template的定义。

#四、Handle class

  1. 使用pimpl idiom方法的class，往往被称为Handle class。
  2. Handle class的实现通常有两种方法：
    • pimpl idiom方法
    • Interface class（接口继承、多重继承）

#五、Handle class的影响

  1. Handle class 解除了接口和实现之间的耦合关系，从而降低文件间的编译依存性。

  2. 对于pimpl idiom方法，成员函数必须通过implementation pointer取得对象数据，会带来如下副作用：

    • 对象消耗的内存数量会增加implementation pointer的大小，这通常不会带来什么问题
    • implementation pointer必须初始化，这会让类承受动态内存分配带来的额外开销，以及遭遇bad_alloc异常（内存不足）的可能性。

  3. 对于Interface class，由于每个函数都是virtual，会带来如下副作用：

    • 每次函数调用都会产生一个indirect jump
    • Interface class的派生对象内含一个vptr指针，会增加对象的内存消耗

#六、使用
以渐进方式使用这些技术，当它们导致的速度和内存占用差异过于重大，以至类之间的耦合不再那么重要时，可以考虑用concrete class替换Handle class.
