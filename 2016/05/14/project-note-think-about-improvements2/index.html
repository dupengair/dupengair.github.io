#关于QueueHelper

  1. Intput()方法中输入参数buffer在后面Line 270时被用到，将内容拷贝到Element新开的Dynbuffer中，程序没有判断这个输入的buffer是否为NULL，此处有bug。

  2. GetRef()中

    if (Iterators.size() > 0 && Iterators.find(iter_id) != Iterators.end())
        {
            Iter = iter;            // ?
            iter_found = true;
        }

Iter = iter;应该取得当前iter_id对应的Iterator才对，觉得要改成Iter = Iterators[iter_id]

#关于errno
通过函数返回值和系统全局错误标识errno共同决定运行结果，而不是单一的返回值

#关于PlatformHelper

  1. GetCpuLoad()方法计算的CPU负载不标准，方法里计算了/proc/stat/cpu%d的0-7这8个数字，实际上只有0-3是cpu的，这样算出来的会显得cpu负载更重

0:user 1:nice 2:system 3:Idle 4:iowait 5:irq 6:softirq 7:stealstolentime 8:guest

CPU利用率 = 100 *（user + nice + system）/（user + nice + system + idle）

#关于NetHelper

  1. StrToIp()接口实现中，调用inet_addr()返回-1时强制改为0的做法不合理，-1可能是全1地址的返回结果，应该先判断地址不为全1再处理inet_addr()的返回值。
  2. CheckNetworkAdapterData()方法中出现MAGIC Number

#锁的改进

  1. Edge OS里面的锁有天生缺陷，没考虑到RAII对象被复制的情况，对此在Effective C++中条款14明确指出处理办法，详见笔记。
  2. 对锁等系统资源，可以用一个计数器基类加以管理，见ME26.
  3. Edge OS里把每种锁的操作方式定义成一个类，把具体的锁作为类的参数，通过不同的宏定义来调用不同的锁。
    • 个人觉得可以用一个锁操作的虚基类通过接口继承实现不同类别锁的操作，这样可以在调用时通过多态的方式，根据传入的不同锁对象自动调用不同的锁方法。

#进程
在Framework中执行进程时调用OS_Process::Execute()接口只是简单的fork一个进程后用exec函数执行指定的程序，没有返回错误的机制，也没有为执行的应用程序提供管的共享的机制。

#接口

  • Framework对内对外使用同一套接口：

    IFrameworkBase -> IFramework -> CFramework
    • 对内：CLoader调用CFramework的接口
    • 对外：在模块中定义指向基类的指针Core::IFramework* FrmInterface;

  • 这样在模块中也可以操作Framework的所有接口

  • 这样的做法不安全，因为不小心或者恶意的模块写法可能导致Framework内部使用的接口被误用，
    可以采用Flyweight设计模式，用继承将内外接口分离，使模块只能使用外部接口。而避免仅以
    简单的指针方式粗暴操作基类，导致模块行为不可控

#C++11

  • throw() 被 noexcept 代替，Agent 和 AudioManager会出错，都有3rd-party使用

#关于模块中的autoconfig

  1. 对于supervisor模块，autoconfig创建res的过程比较不合理，效率低，而且每个资源对象存在一个副本，分别在Framework中和模块自身中维护这个资源对象。

  2. 调用CreateNewRescouce()创建资源时，先要判断其依赖的资源有没有被创建，在Create中创建资源时又进行判断，两处判断对程序没有太多作用，可以先创建所有资源，再构造依赖关系图，创建资源完毕后进行统一的依赖关系判断。

#关于模块中的FwVersion

  1. 模块中有个函数GetFrameworkVersion(FrmwVersion)，作用是取得当前的Framework版本，系统启动运行loadModules会调用这个函数，从模块中取得Framework版本，问题是当前Framework运行为啥要从模块中获得版本，自己不就有版本么，不解。

#关于status和listener
1、EdgeOS里面listener和Notify是简单的回调函数机制实现的，类似内核的NotifyChain技术，考虑用Observer模式来设计，会不会更好一点呢？
