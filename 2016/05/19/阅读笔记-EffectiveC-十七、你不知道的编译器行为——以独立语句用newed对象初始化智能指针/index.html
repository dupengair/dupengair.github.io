看了这一条，只能说，作者在泄露天机——编译器的工作内幕。

一、以对象管理资源

如：
/*声明*/
int priority();
void processWidget(boost::shared_ptr<Widget> pw, int priority);

/*调用*/
processWidget(new Widget, priority());        //error! shared_ptr类型不能隐式转换

  • 这里通不过编译，因为shared_ptr的构造函数是explicit的，不能进行隐式的类型转换，所以我们必须显示转换一下，如下：

    processWidget((boost::shared_ptr<Widget>)(new Widget), priority());    //小心!

二、编译器的工作内幕

  1. 上述改动能通过编译，可以正常运行，但存在内存泄漏的危险.

    • processWidget()函数体在执行前，需要做三件事：

      • new Widget构造一个Widget对象
      • 调用boost::shared_ptr的构造函数初始化这个智能指针
      • 执行priority()函数返回一个int对象

    • 上面的顺序是我们期待的执行顺序，但实际上编译器的处理不是这样，这和C#、Java不同。

    • 可以确定new Widget的执行一定先于调用boost::shared_ptr的构造函数，因为它要作为后者的实参。但priority()函数的执行就不确定了。于是，可能发生这样的执行顺序：

      • new Widget构造一个Widget对象
      • 执行priority()函数返回一个int对象
      • 调用boost::shared_ptr的构造函数初始化这个智能指针

    • 于是，危险发生了，如果执行priority()函数发生异常，意味着processWidget()函数将无法正常执行下去，那么第一步new Widget返回的指向Widget对象的指针将会丢失，因为资源创建后没有被管理对象接管，此时引发了资源泄漏!

三、以独立语句用newed对象初始化智能指针

  1. 要避免上述的资源泄漏很简单，就是以独立语句用newed对象初始化智能指针。

    如：
boost::shared_ptr<Widget> Wptr(new Widget);        //先初始化智能指针
processWidget(Wptr, priority());            //这时就不存在资源泄漏的危险了

四、再论编译器

  • 上述修改之所以行得通，是因为编译器对于跨语句的操作没有决定优先权的自由度，只能在语句上进行重新排列。
