八、天生的矛盾体——析构函数与异常处理

May 28, 2016 | 阅读笔记EffectiveC++ | Hits
文章目录
  1. 1. 一、析构函数绝对不要抛出异常
  2. 2. 二、将产生异常的操作转移到客户端进行处理，给客户提供一个接口

既生瑜（析构函数），何生亮（异常处理）……
一山难容二虎……

在C++中，只要析构函数抛出异常而没有处理导致异常传播出去，程序可能过早结束或出现不明确行为，特别是多个异常同时存在的情况下。因为析构函数会在抛出其它异常的过程中被调用，如果析构函数抛出异常，这个新的异常可能会在现存的异常到达catch子句中被抛出，导致程序调用terminate()。

一、析构函数绝对不要抛出异常

  1. 如果一个被析构函数调用的函数可能抛出异常，析构函数中应该捕捉任何异常，然后吞下或结束程序，决不能传播出去。

    如：
class A
{
public:
    static A create();
    ……
    void close();        //操作失败可能抛出异常
}

class B
{
    public:
    ……
    ~B() { a.close(); }    //析构时操作A对象的close()方法可能产生异常，如果这里不做处理，异常会传播出去，也就离开了这个析构函数
private:
    A a;
}

  2. 如果抛出异常就结束程序，可以在析构函数中调用abort处理

    如：
B::~B()
{
    try {a.close();}
    catch(...)
    {
        std::abort();        //强迫结束程序，防止异常从析构函数中传播出去，造成不明确行为
    }
}

  3. 如果不结束程序，就在析构函数中吞下异常

    如：
B::~B()
{
    try {a.close();}
    catch(...)
    {
        ……;        //吞下异常，防止异常从析构函数中传播出去，造成不明确行为
    }
}

二、将产生异常的操作转移到客户端进行处理，给客户提供一个接口

  1. 如果某个操作可能抛出异常，而又必须处理这种异常，那么这个异常必须来自析构函数以外，因为析构函数抛出异常就是危险。

  2. 给客户提供一个响应异常的机会

    如：
class B
{
public:
    void close()                    //提供给客户端的接口，让客户有机会处理异常
    {
        a.close();
        closed = true;
    }
    ~B()
    {
        if(!closed)
        {
            try { a.close(); }        //如果客户没有处理，才在析构函数中处理
            catch ()
            {
                ……;                //吞下异常，防止传播
            }
        }
    }
private:
    A a;
    bool closed;
}
