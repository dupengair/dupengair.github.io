二十四、了解vitrual functions、multiple inheritance、virtual base classes、RTTI的成本

May 28, 2016 | 阅读笔记MoreEffectiveC++ | Hits
文章目录

  2. 2. 二、成本二：在每一个拥有虚函数的对象内付出“一个额外指针”的代价
  3. 3. 三、成本三：放弃inline
  4. 4. 四、多重继承
  5. 5. 五、 RTTI
  6. 6. 六、总结

非虚函数——包括必定是非虚函数的constructors——会像一般的C函数那样被实现

一、成本一：必须为每个拥有虚函数的class耗费一个vtbl空间，其大小视虚函数的个数（包括继承的）而定

  1. class的vtbl被产生于“内含第一个non-inline，non-pure虚函数定义”的目标文件中。
  2. 避免将虚函数声明为inine，否则编译器会在每一个使用了vtbl的目标文件中产生一个vtbl的复制品，目前的编译器通常会忽略inline标识

二、成本二：在每一个拥有虚函数的对象内付出“一个额外指针”的代价

  1. 较大的对象意味着较难塞入一个缓存分页或者虚内存分页中，也意味着换页活动会增加，性能会降低
  2. 对于虚函数调用，编译器会产生代码，完成以下动作：
    • 根据对象的vptr找出vtbl。成本只有一个偏移调整，以便获得vptr和一个指针间接动作，以便获得vtbl。
    • 找出被调函数在vtbl内的对应指针，成本是一个偏移，以进入vtbl数组
    • 调用该虚函数
  3. 因此，调用一个虚函数的成本，基本上和通过一个函数指针来调用函数相同，虚函数本身不构成性能上的瓶颈

三、成本三：放弃inline

  • 虚函数不应该inline，因为inline意味着编译期对调用端进行扩展，virtual则意味着运行期决定被调函数，两种相互矛盾

四、多重继承

  1. 一个对象之内有多个vptrs，每个base class一个，针对base classes形成的特殊vtbls也被产生出来。虚函数对每一个object和每一个class的空间负担又有所增加，运行期成本有轻微增长。
  2. virtual base class的实现，常常利用指针指向公共的基类成员，以消除复制行为。这可能导致对象内隐藏指针增加。
  3. 对小型classes而言，额外开销的相对量会比较大，对大型class则无足轻重。

五、 RTTI

  1. 运行时objects和classes的相关信息被存放在类型为type_info的对象内，可以利用typeid操作符获取某个class对应的type_info对象。
  2. 只有当某个类型拥有至少一个虚函数，才保证我们能够检验该类型对象的动态类型。
  3. RTTI的设计概念是：根据class的vtbl来实现
  4. RTTI的空间成本只需在每一个class vtbl内增加一个条目，在加上每个class所需的一份type_info对象空间

六、总结

  • 大部分情况下，相对于编译器生成的代码，手动编写的可能比较没效率，也不够健壮。
