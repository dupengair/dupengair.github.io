十四、明智运用exception specifications

一、exception specification

  1. 事实：如果函数抛出一个未列于exception specification的exception，这个错误会在运行期间被检查出来，特殊函数unexpected会被调用。
  2. unexpected的默认行为时调用terminate，terminate的默认行为是调用abort，所以程序如果违反exception specification，默认结果就是程序被终止。
  3. 编译器只会对exception specification做局部性校验，C++不允许编译器校验函数内部调用某个可能违反调用端函数本身exception specification的情况。
  4. 如果带有exception specification的新代码和缺乏exception specification的旧代码要整合到一起，这种弹性就有必要。

二、关于template

  1. 避免将exception specification放在需要类型自变量的templates身上
  2. 没有任何一个方法可以知道一个template的类型参数可能抛出什么exceptions，所以不要为template提供exception specification
  3. 不应该将exception specification与template混用

三、关于回调函数

  1. 如果A函数内调用了B函数，而B函数无exception specification，那么A函数本身也不应设定exception specification。
  2. 要特别注意注册回调函数的情况

四、处理系统可能抛出的exceptions

  1. 记得处理系统可能抛出的exceptions
  2. 最常见的是bad_alloc，发生在operator new在分配内存失败时抛出

五、处理非预期的exceptions

  1. C++允许以不同类型的exceptions取代非预期的exceptions

    class UnexpectedException();

void convertUnexpected()
{
    throw UnexpectedException();
}

set_unexpected(convertUnexpected);
    • 只要被违反的exception specification内含UnexpectedException，exception就能继续传播并继续下去。但如果exception specification未包含UnexpectedException，terminate就会被调用，像从未取代unexpected一样。

  2. 将非预期的exceptions转换为一个已知类型可基于一个事实：如果非预期函数的替代者重新抛出当前的exception，该exception会被标准类型bad_exception替代

    void convertUnexpected()
{
    throw ;
}

set_unexpected(convertUnexpected);
    • 此时任何非预期的exception都会被一个bad_exception替代，并且传播下去。

六、局部性校验

  1. 局部性校验导致编译器只校验函数本身exception specification，不校验其内部调用的其它函数
  2. 这会造成，当一个较高层次的调用者已经准备好处理发生的exception时，unexpected函数却被调用。
