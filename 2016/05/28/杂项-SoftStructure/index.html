一、架构

  • 对现有重复业务的抽象 -> 行业经验
  • 对未来业务拓展的前瞻 -> 对未来业务发展的预见能力

确定系统边界

一门分解和连接的工艺

二、维度

  • 技术架构

    • 解决与业务无关的问题
    • OS Wrapper、Platform Wrapper、中间件、消息通信、日志、系统资源管理、灾难恢复、升级、数据持久化、文件系统、编译环境

  • 业务架构

    • 解决业务问题
      • 模块级：按照业务功能进行模块划分（如视频、录像、事件）
      • 系统级：多个服务组成的大型系统（如人脸识别：前端+服务器）

四、思路

  1. 方案平台轻量化（CC）
    • 通过CC抽象出一致的软件调用接口，屏蔽不同平台的差异
    • 平台变化不用更改应用层代码
    • 封装平台内部的处理逻辑，对应用程序不可见

  2. 软件设计服务化（去中心化，组件式开发，软件分层）

    • 去中心化

      • 不依赖于中心调度
      • 关键服务独立进程，增强系统鲁棒性
      • 服务之间以消息通信，消除依赖
      • 可以自由裁剪，可伸缩性强
      • 提高可测试性

    • 组件式开发

      • 提供系统公共组件，提供一致的实现与调用方式
      • 不实现具体业务，实现功能的抽象，应用程序在其基础上进行开发
        • 配置参数
        • 调用接口
        • 注册回调
      • 稳定的组件带来稳定的应用程序
      • 业务实现与核心代码分离，易于维护
      • 组件开发注重代码重用、效率、稳定性
      • 业务逻辑代码注重功能、需求，不考虑代码重用
      • 功能组件举例：
        • buffer管理、通信机制、调试接口、线程模型、事件管理、资源管理、系统日志

    • 软件分层

      • OSWrapper
        • fs

        • …

        • logs module
        • …
      • 业务构成(application/business)
        • tools
          • watchdog

  3. 编译环境生态化（独立文件系统、统一调试环境、方案平台兼容性）

    • 可生长可变化，方便定制与剪裁
    • 独立文件系统
      • 不依赖平台
      • 设备特性统一
      • 方便调试/测试
    • 统一调试环境
      • 调试和测试工具
      • 日志输出与关键信息过滤
      • 异常状态甄别
      • 调试方法
    • 方案平台兼容性
      • 降低编译环境对sdk依赖度
      • 方便新平台引入和SDK更新

六、方向

  1. 完全是一个单片机程序，没有模块与层次划分、缺乏系统资源管理
  2. 只具有函数级改良的价值，不具有复用价值，所有代码重构
    • 代价最大
  3. 完全不修改，直接在Hisi平台使用
    • 稳定，不需要改动
    • 代价最小
  4. 只考虑多平台复用，不做大改动
    • 引入CC，配合修改视频相关的代码逻辑
    • 至少修改MIB为可读性强的数据结构(XML、Json…)
    • 以后有问题再不断重构
  5. 引入CC的基础上，做模块化
    • 引入OS中间件，屏蔽操作系统复杂度，统一进行资源管理
    • 模块级重构
    • 功能定义与配置描述
    • 消息通信
  6. 模块化的基础上完善资源管理与调试、日志
    • 前提是完成模块化并具有OS中间件
    • 统一资源管理、调试方法、日志操作
  7. 完善系统调优
    • 系统性能调优
    • 模块级代码优化
  8. 性能评估
    • 特定环境
    • 特定工具
    • 指标与标准
