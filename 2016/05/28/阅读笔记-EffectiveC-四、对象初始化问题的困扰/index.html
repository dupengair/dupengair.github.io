所谓对象初始化问题，就是确定对象在使用前已被初始化，即ATTI技术，我们都知道读取未初始化的值会导致不明确的行为。

一、C++中对象的初始化

  1. C part of C++，且初始化可能招致运行期成本，不保证初始化，如array。
  2. non-C part of C++，可以初始化，如vector（STL part of C++）。

二、确保每一个构造函数将对象的每一个成员初始化

  1. C++规定，对象的成员变量初始化发生在进入构造函数本体之前。
  2. 如果没有初始化成员列表，会先调用default构造函数来初始化对象成员。
  3. 因此，构造函数体中进行赋值操作前，各成员变量已经经历了初始化，构造函数体中的赋值操作不是初始化。
  4. 因此此，构造函数体中的赋值操作带来高成本，因为default构造函数的操作被浪费了。
  5. 因此此此，初始化列表技术比构造函数体中对成员变量的初始化更高效，总是采用初始化列表是个好习惯。
  6. 初始化列表中对各个成员变量设的实参，作为调用各个成员变量构造函数的实参。

三、初始化列表总是列出所有成员的初始化

  1. 对于非内置类型，初始化列表效率高，上面已讨论。
  2. 对于内置类型，初始化和赋值的成本相同，为了一致性，也最好用初始化列表。

  3. 想要用初始化列表default构造一个成员变量，指定一个nothing的初始化实参，如：

    ABEntry::ABEntry():theName(),theAddress(),thePhones() {};    //ABEntry的构造函数

四、何时必须要初始化列表

  1. 如果成员变量是const或references，一定要初始化，不能被赋值。
  2. 总是使用初始化列表，往往更高效。

五、关于初始化顺序

  1. 基类总是比派生类更先初始化。
  2. 类内部的成员变量以声明的顺序初始化（注意不是初始化列表的顺序）。
  3. 因此，在初始化列表中最好以类成员的声明顺序。

六、不同编译单元内定义之non-local static 对象的初始顺序

  1. static 对象：生命期从被构造出来直到程序结束。包括global对象，作用域中声明为static的对象，不包括stack和heap-based对象。
  2. local static对象：函数中的static对象。non-local static对象：其它对象，程序结束时析构函数在main()结束时被自动调用。
  3. 编译单元：产出单一目标文件的源码，源码文件+头文件。

七、问题

真正的问题是：如果某编译单元内的某个non-local static对象的初始化动作使用了另一个编译单元的某个non-local static对象，它所用到的这个对象可能尚未初始化。C++对“定义于不同编译单元内的non-local static 对象”的初始化次序无明确定义。

  • 《源文件1：程序库》

    class FS{
    public:
    std::size_t numDisks() const;
    };
extern FS tfs;            //non-local static

  • 《源文件2：客户端程序，使用程序库中的global 对象》

    class Dir{
    public:
    Dir(params);
    };
Dir::Dir(params)
    {
    std::size_t disks = tfs.numDisks();        //用到上面的non-local static
    }
Dir tempDir(params);        //disaster!
    • tfs与tempDir在不同的源码文件中建立，是定义于不同编译单元内的non-local static对象，不能保证在建立tempDir对象时tfs对象已经初始化。

八、解决初始化次序不确定性的办法

  1. 将每个non-local static对象搬到自己的专属函数中，该对象在此函数中被声明为static，这些函数返回一个reference指向所含的对象。然后用户调用这些函数，不直接操作对象，换句话说，用local static替换non-local static。这是Singleton模式的常见实现方式。
  2. C++保证，在函数内的local static对象会在“该函数调用期间”“首次遇上该对象的定义”时被初始化。

  3. 不调用这些专属函数，就不会引发local static对象的构造与析构成本。

    • 《源文件1：程序库》

      class FS{
    public:
    std::size_t numDisks() const;
    };
FS& extfs()            //static 对象专属函数
    {
    static FS fs;
    return fs;
    }

    • 《源文件2：客户端程序，使用程序库中的global 对象》

      class Dir{
    public:
    Dir(params);
    };
Dir::Dir(params)
    {
    std::size_t disks = tfs().numDisks();        //使用指向static 对象的references
    }
Dir& tempDir(params)        //static 对象专属函数
    {
    static Dir td;
    return td;
    }

九、使用static 专属函数时的注意事项

  1. 这种结构下的static 专属函数很单纯，创建static对象并且返回它的引用，非常适合inline。
  2. 内含static对象的函数在多线程系统中有不确定性
  3. 不适合同时被多个使用，见条款21

十、总结

为避免在对象初始化前过早使用：

  1. 手动初始化内置对象
  2. 使用初始化列表处理所有对象
  3. 对于“初始化次序不确定性”，加强设计
