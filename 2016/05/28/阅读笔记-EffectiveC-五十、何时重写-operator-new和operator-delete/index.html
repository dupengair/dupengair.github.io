五十、何时重写 operator new和operator delete

May 28, 2016 | 阅读笔记EffectiveC++ | Hits
文章目录
  1. 1. 一、常见的原因
  2. 2. 二、一个初级版本的operator new实例
  3. 3. 三、内存对齐（alignment）
  4. 4. 四、使用自定义operator new和operator delete 的目的

一、常见的原因

  1. 检测运用上的错误

    • delete失败导致的内存泄露
    • 对同一段内存的多次delete导致不定义行为
    • 利用签名进行内存溢出检测

  2. 强化效能

    • 编译器的operator new和operator delete主要用于一般目的：
      • 寿命（长时存在、短时存在）
      • 大小（大块、小块、大小混合）
      • 破碎问题
    • 这使得默认的实现对每种情况都适度的好，但不对特定情况有最佳表现。为特定情况定制可能获得重大效能提升——多快好省的使用内存。

  3. 收集使用上的统计数据

    • 大小分布
    • 进出次序
    • 运用形态
    • 最大动态分配量

二、一个初级版本的operator new实例

  • 这个实例主要体现检测overruns和underruns

    static const int signature = 0xDEADBEEF;
typedef unsigned char Byte;

//这段代码存在若干bug
void* operator new(std::size_t size) throw(std::bad_alloc)
{
    using namespace std;
    size_t realSize = size + 2 * sizeof(int);        //增加大小用以插入signatures

    void* pMem = malloc(realSize);                    //调用malloc获取内存
    if(!pMem) throw bad_alloc

    *(static_cast<int *>(pMem)) = signature;        //插入signatures
    *(reinterpret_cast<int *>(static_cast<Byte *>(pMem)
        +realSize-sizeof(int))) = signature;

    return static_cast<Byte *>(pMem) + sizeof(int);    //返回指针指向第一个signature之后的位置
}

  • 这段代码有两个主要缺陷，目的在于演示如果自定义operator new：

    • 疏忽了身为这个特殊函数所应具备的“坚持C++规矩”的态度——所有的operator new 操作符都应该包含一个循环，在这个循环中反复调用new-handling函数，如同默认operator new的行为。
    • 内存对齐（alignment）

三、内存对齐（alignment）

  1. 许多计算机体系结构要求特定的类型必须放在特定的内存位置
  2. C++要求所有operator new返回的指针都适当的对齐（取决于数据类型）。

  3. malloc就是这样工作的，所以让程序返回一个来自于malloc的指针是安全的。

    • 所以上面实例中，自定义operator new返回一个来自于malloc并偏移int大小的指针，这是不安全的。如果客户调用这个操作时企图给double类型分配内存，这时候内存未能适当对齐，会导致程序崩溃或执行变慢的未定义行为。

  4. Boost库的Pool分配器对最常见的“分配大量小型对象”很有帮助

  5. TR1支持各类型的对齐条件

四、使用自定义operator new和operator delete 的目的

  1. 为了检测运用出错（如前）
  2. 为了收集动态分配内存的使用信息（如前）
  3. 为了增加分配和归还速度
    • 泛用型分配器往往（虽然并不总是）比定制分配器慢，特别是当定制性分配器专门针对某些特定类型对象设计时。Class专属分配器是“固定大小内存”分配的实例。Boost提供的Pool也是。不过在设计之前，首先应分析程序，确认程序瓶颈的确发生在那些内存函数上。
  4. 为了降低缺省内存管理器带来的空间额外开销
    • 泛用型分配器往往不只比定制的慢，而且使用更多内存，因为它们常常在每一个分配区块上招来额外开销。针对小型对象开发的分配器如Boost库的Pool，本质上消除了这些额外开销。
  5. 为了弥补缺省分配器中的非最佳内存对齐方式（suboptimal alignment）
    • 如果固定访问某个大小对齐的类型，定制一个operator new来保证类型内存对齐，可能大幅提高程序效率。编译器自带的operator new并不保证动态分配的内存对该类型对齐。
  6. 为了将相关对象集中
    • 如果特定的某个数据结构往往被一起使用，希望在处理这些数据时将“内存页错误（page fauls）”频率降至最低，那么为这个数据结构创建另一个heap就有意义。这样可以讲它们集中在尽可能少的数据页上。new和delete的“placement”有可能完成这样的行为。
  7. 为了获得非传统行为
    • 比如希望分配和回收共享内存（memory map）的区块，但唯一能管理该内存的只有C API函数。或者自定义一个operator delete将回收的内存内容覆盖为0，以增加应用程序安全性。
