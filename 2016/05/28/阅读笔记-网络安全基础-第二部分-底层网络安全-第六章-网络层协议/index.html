一、网络层标准

  1. 端到端传输
    • 用于专用网
    • 通常看作设备间的点对点连接
  2. 网际协议
    • IPv4
      • 每个设备一个网络层负责源到目标的路由
      • IP地址与网络的物理位置或互连是无关的，因此互联网采用分布式的路由方法，基于每一台设备知道数据包需要送达的下一个目标
      • IP协议格式：
        • ID字段用于支持分片和重组
        • 所有分片的长度，除了最后一个字段，都必须能被8整除，因为偏移量x8标识了分片在重组缓冲区里的位置
    • ARP
      • ARP设计的目的是用于不同的物理网络层
      • 可用于确定两台设备是否有相同的IP地址
        • 发送ARP请求，询问IP地址对应的硬件地址
        • 收到应答则说明有地址冲突
      • 利用ARP应答使发送者把数据包发送到错误的地址，称为ARP欺骗或ARP缓冲区中毒
      • 如果攻击者能公开扫描一个网络中少量主机的地址，将产生数量巨大的ARP请求包
    • ICMPv4
      • ICMP是IP协议的组成部分，起辅助IP的作用，作为载荷携带在IP数据包中
      • ping请求使用一个对一组ping唯一的标识号
      • 路由器使用重定向消息通知本地同一网络的主机，是否有到达目标更好的路由存在
      • ICMP消息也被攻击者用来确定一台主机是否在运行
    • 基于头部的攻击：
      • 引起最大麻烦的是长度、标志和偏移量
      • 从安全角度可以将IP头部分为两类：
        • 端点字段：
          • 由端点使用的字段构成，在传输过程中不检测
          • 包括长度、标识、标志、偏移、协议和源IP
          • 对端点的攻击如死亡之ping、将源设为和目标地址相同、将源地址设为广播等
        • 传递字段：
          • 由各个路由器进行检测，并在传输过程总可能被修改
          • 攻击通常引起路由器对数据包丢弃
      • 极少有对ARP和ICMP的头部攻击
        • 对ARP头部的攻击必需与目标处在相同网络，且无效的ARP数据包通常被丢弃
        • ICMP头部很简单
    • 基于协议的攻击：
      • IP和ICMP协议比较简单，因为通过网络传递数据的设备间没有数据包的交换。大多数针对IP和ICMP的攻击瞄向数据包的路由，设法引起数据包错误路由
      • 攻击者可以使用ICMP错误消息引起拒绝服务，或者将流量重定向到错误目标。这样的攻击要求攻击者能沿路径嗅探流量以发现IP数据包，根据其头部信息产生一个ICMP错误消息
      • ARP的常见攻击是，一个ARP请求被攻击者发现，并在正常的回答前产生无效的ARP应答，造成目标的ARP缓存信息不正确，其发出的数据包将送到错误的目的主机
    • 基于验证的攻击：
      • IP地址是识别互联网设备的唯一标识，应用程序提供服务前使用IP地址作为验证设备的方法
      • 创建一个源地址与自身IP不相同的数据包是有可能的，即IP地址欺骗
        • 攻击者使用一个欺骗性IP地址向网络发送一个ICMP回应请求，会引起目标计算机向受害者发送ICMP应答，甚至可以直接发送IP广播
      • 大多数路由器可以配置为检查直连网络上数据包的源IP，能阻止源IP与对应网络不一致的数据包外发
    • 基于流量的攻击：
      • IP嗅探通常被用于网络安全，监控流量发现是否包含攻击，设备通常被部署在网络边界点
      • APR风暴:
        • 对无效IP的ARP广播可能多次重复
        • 根据子网内主机数量，少数攻击者能制造大量请求的广播包
        • 通过路由器上禁止ICMP回显请求可以避免
      • IP层允许向目标网络或主机发送数据包，可能导致流量无法控制，一般来说针对流量的攻击难以避免
  3. DHCP协议
    • 协议
      • 支持同一个设备总是得到相同的IP
      • 有静态池和动态池，动态池需要续租
      • 客户端在到期时间的50%时进行续租，成功重设租期，失败在87.5%再次续租
      • 客户端可以在任何时间放弃IP
    • 基于头部的攻击
      • 作为UDP数据包的载荷，因此没有任何基于头部的攻击(?)
    • 基于协议的攻击
      • 使用伪造硬件地址寻租，造成动态池IP耗尽
      • 伪造客户端释放IP，造成IP被租给其它客户端，使得网络出现IP冲突
    • 基于验证的攻击
      • 冒充DHCP服务器对客户端请求进行应答，使客户端使用无效地址，通信发发生问题
    • 基于流量的攻击
      • 对DHCP来说，流量嗅探和雪崩式攻击都不是主要关注点
  4. IPv6协议
    • 协议
      • 基本头部的设计供基本路由使用，目的是最小化路由器所需的计算量来加速路由。扩展的头部供两端设备使用
      • 优先级确定怎样处理同源的多个数据包
        • 网络要拥塞导致最低优先级的数据包被丢弃
        • 0~7保留给上层协议可以为丢弃的数据包补偿的流量
        • 8~15保留给上层协议对丢弃的数据包不重传的流量，通常用于音视频这样的实时协议
        • 流标记和源IP地址创建经过路由器的唯一流
    • 与IPv4安全问题相同
  5. ICMPv6
    • 格式与ICMPv4相同，数据包类型减少
    • 查询包类型也减少，去掉时间戳和地址掩码请求消息，增加了ARP作为ICMP相邻请求和广播消息

二、常用对策

  1. IP过滤

    • 阻止ICMP应答以防止IP在线被侦测
    • 阻止除DNS外的UDP
      • 基于端口的过滤并不完善，欺骗性应用可以使用合法端口
    • 基于IP过滤的通常作为路由器第一道防线，依据恶意地址进行过滤，但无法防止攻击者转移

  2. 网络地址转换(NAT)

    • 在公网和私网地址范围进行转换，从而达到大量设备共享少量公共地址的目的
      • 对私网，NAT设备是一台路由器，对公网，NAT设备是最终的目标地址
      • 静态NAT一对一，动态NAT一对多
      • 可以将公私网络隔离，以提高安全性
    • NAT的一个问题是第一个数据包从公网进入，映射表上没有任何条目时
      • 隧道背后的思想是将一个公网IP和端口与一个私网IP和端口进行映射，当数据包到达NAT上一个已被隧道化的端口时，NAT接收数据包并重写IP和端口，根据映射表将数据包发送到私网主机
      • 隧道的局限是，如果使用公网IP和端口连到多个私网主机，每个公网IP和端口只能与一个私网主机进行组合
    • NAT可以被配置为“传递”
      • 私网中的NAT和主机同时拥有公网和私网地址
      • 对网络上的计算机访问不受隧道限制
      • 接入公网的计算机受到攻击时，可以通过增加私有IP访问私网，这样的配置是不安全的

  3. 虚拟专用网(VPN)

    • 提供双方设备间的加密和验证通信信道，在IP层可分为三类：
      • 网络到网络
      • 客户端到客户端
      • 客户端到网络
    • 应用
      • 连接两个网络
        • 两个网路各有独立的地址范围
        • VPN提供两个VPN节点间的加密和连接验证
      • 使远程网络成为主网络的扩展
        • 主网络能控制和监控子网络的流量，为其提供安全
      • 客户端到客户端
        • 实现与远程VPN服务器的通信
        • 两个设备间的所有流量是加密的，连接需要验证
      • 客户端到网络
        • 提供到主网络的访问，使远程客户端通过VPN连接到主网络
        • 远程客户端有两种配置：
          • 远程客户端有两个IP，用于接入互联网的地址，以及和自己所在主网络的地址，到主网络的流量使用VPN，其它流量使用互联网
          • 来自客户端的流量全部进入主网络，主网络让访问互联网的流量通过
    • VPN有助于避免嗅探和验证

  4. IPSec

    • IPSec是为IPv6设计的支持加密和验证的协议，能用于VPN协议
    • 能减少嗅探和验证攻击，在VPN中运行良好。问题是跨公网的秘钥分发，需要公共秘钥架构PKI的支持
    • 使用两个扩展头部，一个支持验证，一个支持加密和验证，用于IPv4时是载荷的一部分，因为不支持扩展头部

    • 验证头部(AH)：

      | 基本头部(40B) | 载荷(65535B) | 验证头部(AH) | 上层载荷 |
| <-                        验证的                    -> |    
验证头部AH：
| 下一头部 | 头部长度 | 保留的 | 安全参数索引 | 序列号 | 验证数据 |
      • 对整个数据包进行hash，然后用安全秘钥对hash值加密
      • 安全参数索引：区分同一数据流中所有数据包
      • 序列号：防止数据包的重发
      • 验证数据：存储加密的hash值

    • 支持加密和验证的头部：

      | 基本头部(40B) | 载荷(65535B) | ESP头部 | 上层载荷 | ESP拖挂 | 验证数据 |
               | <-                 验证的              -> |
                              | <-         加密的       -> |

ESP头部： 
| 安全参数索引 | 序列号 |

ESP拖挂：
| 打补丁 | 补丁 | 补丁长度 | 下一头部 |
      • 载荷是由安全秘钥和ESP尾部一起加密的
      • 验证数据时ESP头部、载荷和ESP拖挂的hash值
