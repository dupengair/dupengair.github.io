#UNIX域套接字

一、概述

  1. UNIX域支持套接字
    • 字节流套接字
    • 数据报套接字
  2. 用途
    • 比TCP套接字快一倍
    • 可以传递描述符
    • 可以传递用户凭证(用户ID和组ID)，以提供额外的安全措施
  3. Unix域用文件系统的路径名来标识客户端和服务端的协议地址
    • 需要bind到套接字上
    • 如果路径已存在，会bind失败，需要先unlink路径名

二、套接字地址结构

#include <sys/un.h>
struct sockaddr_un {
    sa_family_t sun_family;
    char        sun_path[14];
};
  • sun_path数组中的路径名必需以空字符结尾

三、socketpair

#include <sys/socket.h>
int socketpair(int family, int type, int protocol, int sockfd[2]);
成功返回0，失败返回-1
  • 创建两个随后连接起来的套接字，只用于Unix域套接字
  • family必需为AF_LOCAL，protocol必需为0
  • type可以为SOCK_STREAM或者SOCK_DGRAM
  • 新创建的描述符作为sockfd[0]和sockfd[1]返回
  • 这样得到的结果为流管道，与PIPE的区别为流管道是全双工的
  • 书上说Berkeley的实现是半双工(？)

四、套接字函数

  1. bind绑定后的路径名默认权限是0777，并按照当前umask值进行修正
  2. 路径名需要为绝对路径，不然对端就需要知道本端的应用程序路径才能成功通信
  3. connect指定的路径必需是已绑定的Unix域套接字且已打开，type也需要一致，错误条件如下：
    • 路径未绑定到套接字
    • 路径绑定的套接字未打开(没监听)
    • 监听的套接字类型不符
  4. connect连接域套接字对路径的权限测试，相当于open写打开
  5. Unix域字节流套接字类似TCP
    • 提供无记录边界的字节流服务
    • 监听队列满时，立即返回ECONNREFUSED错误（注意TCP套接字上，监听队列满时，忽略新到达的SYN，等待客户端重试）
  6. Unix域字数据报套接字类似UDP
    • 提供有记录边界的不可靠数据报服务
    • 在未绑定的Unix域套数据报接字上发送数据报不会自动绑定一个临时路径(TCP或UDP套接字则会自动绑定一个临时端口进行发送)
    • 所以除非数据报发送端已绑定路径名到其套接字，否则接收端无法发送应答，因为服务端的recvfrom会返回空路径做对端地址，导致sendto错误

五、描述符传递

  1. 自然的传递

    • fork调用返回以后，子进程共享父进程所有打开的描述符
    • exec调用执行以后，所有描述符的打开状态不变

  2. 过程

    • 创建Unix字节流套接字，不推荐数据报套接字，可能被丢弃
    • 可被传递的描述符不限类型(文件、套接字…)
    • 发送进程穿件msghdr结构，通过msg_control成员作为辅助数据，通过sendmsg发送
    • 接收进程通过recvmsg接收消息，这个消息由内核来专门处理
    • 在sendmsg后，即使发送进程在接收进程调用recvmsg前闭了描述符，对于接收进程仍然处于打开状态
    • 接收到的描述符值与发送的不同是正常的，传递描述符的过程是创建新的描述符，指向内核中相同的文件表项
    • 如果接收进程调用recvmsg时没有可用空间，这个被传递的描述符就会被关闭
    • 接收进程调用recvmsg的过程中避免使用MSG_PEEK标志，否则后果不可预料
      • MSG_PEEK：允许查看已可读取的数据，而且系统不在recv或recvfrom返回

六、接收发送者的凭证(uid、pid、gid …)

  1. 用于服务器验证客户端是否有权限请求相应服务

    #include <sys/socket.h>
struct cmsgcred {
    pid_t cmcred_pid;
    uid_t cmcred_uid;
    uid_t cmcred_eupid;
    gid_t cmcred_gid;
    short cmcred_ngroups;
    gid_t cmcred_gourps[CMGROUP_MAX];
};
    • FreeBSD使用
    • CMGROUP_MAX通常为16，cmcred_groups至少为1，cmcred_ngroups第一个元素是有效组ID

广播

一、概述

  1. 寻址方式

    类型  IPv4  IPv6  TCP UDP  标识接口数  递送到接口数
单播    Y     Y    Y   Y      1个         1个     
任播          Y        Y      一组     一组中的一个
多播   可选    Y        Y      一组     全组 
广播    Y              Y       全体       全体
    • TCP只支持单播，UDP和原始套接字支持别的
    • 任播从一组主机(通常提供相同服务)中选一个(通常按照某种测度离源主机最近)
    • IPv4可选多播，支持广播，IPv6必需支持多播，不支持广播
    • 广播的用途：
      • 资源发现，在本地子网定位一个服务器主机
      • 减少网络环境中的分组流通
    • 广播的应用：
      • ARP：使用链路层广播，通过IP找MAC
      • DHCP：客户端广播自己的请求(255.255.255.255)
      • NTP：服务器主机广播当前时间，避免客户端自己的轮询
      • 路由守护进程routed：广播路由表，使附近的路由器或者主机可以接收通告，以更新自己的路由表

二、广播地址

  1. 记法{子网ID， 主机ID}
    • 子网ID表示子网掩码覆盖的连续位
  2. 子网定向广播地址
    • {子网ID，-1}，如192.168.1.100/24对应为192.168.1.255
    • 最好不要依赖转发子网定向广播进行通信，路由器设置程序允许转发此广播包可能造成放大攻击
  3. 受限广播地址
    • {-1, -1}，255.255.255.255
    • 路由器从不转发
    • 主机发送时将该地址转换成定向子网广播地址
    • 如果发送端有多个接口，可能只在第一个接口上发，如果要每个接口都发送，应该首先获取每个接口的配置，然后对接口所在的子网执行sendto调用，发送定向广播

三、广播与多播

  1. 以太网帧类型字段，IPv4为0x0800，IPv6为0x86dd
  2. 主机发送时，将广播地址映射成48位全1的以太网地址: ff:ff:ff:ff:ff:Ff
  3. 主机收到广播但UDP未绑定对应端口时，不能发送ICMP端口不可达消息，以避免广播风暴
  4. 广播分组到达子网上所有主机，包括发送主机自身(一般来说，发送主机不用绑定目的端口，不然数据会返回到应用层)
  5. 不关心广播的所有主机沿协议栈相识完整处理UDP广播包，直到被UDP层丢弃
  6. 子网上所有的非IP主机在链路层收到完整的帧，然后丢弃

#多播

一、概述

  1. 单播地址标识单个IP接口，广播地址标识子网内的所有IP接口，多播地址标识一组IP接口
  2. 多播数据报只应由对其感兴趣的接口接收
  3. 多播可以跨网段使用
  4. 禁止源地址为广播或者多播的IP数据报

二、多播地址

  1. IPv4的D类地址

    • IPv4的D类地址(224.0.0.0~239.255.255.255)为多播地址，低28位为多播组ID，整个32位为组地址

    • 映射到以太网链路地址：

          |<-    28b组ID   ->|
  |e:[4b]|[1b]:[7b]| | |        IPv4D类地址
    |<-忽略的  |        |
       5bit ->|<- 低序的|
              | 23bit->|
|01|00|5e|[1b]:[7b]| | |      以太网链路地址

    + 映射到以太网地址高24位总是01：00：5e，下一位总是0
    + 低23位复制多播组ID的低23位，多播组ID的高5位总被忽略，因此32bit多播地址映射为单个以太网地址的关系，不是一对一的
    + 以太网地址的首字节低序两位标明该地址是一个统一管理的组地址，意味着高24位由IEEE分配
* 特殊的IPv4多播地址：
    + 224.0.0.1 所有主机组(all-host)
    + 224.0.0.2 所有路由器组(all-routers)
    + 224.0.0.0~224.0.0.255 链路局部多播地址(link local)
        - 为低级拓扑发现和维护协议保留
        - 多播路由器从不转发这些目的地址的数据报

  1. IPv6的多播地址

    • 映射到以太网链路地址：

                      |<-     112bit组ID      ->|
|ff|  [4b]:[4b] |     ...     |  |  |  |  |      IPv6多播地址
   |<-4:4b范围->|<-10Byte忽略->|<- 低32b ->|     
                |             |<-低序 32b->|
                        |33|33|  |  |  |  |      以太网链路地址
          局部管理的组地址->|
      • IPv6多播地址高字节为ff
      • 112bit组ID的低序32bit复制到以太网地址的低32bit，以太网高2字节为33:33
      • 以太网首字节低序2位表明地址是一个局部管理的组地址，意味着不能保证该地址对IPv6地址的唯一性

    • IPv6多播地址定义

      标志：| 0 | 0 | P | T |

      • 4bit标志字段的高2bit保留

      • 当P=0时

        • T=0：众所周知多播组

        • T=1：临时多播组(transient)

          标志：| 0 | 0 | 0 | T |

          |ff|[4b] : [4b]|    ...   |   |   |   |   |   IPv6多播地址
   |<-4:4b范围->|<-80bit0->|<- 低32b组ID ->|

      • 当P=1时

        • 表示多播地址是基于某个单播前缀赋予
        • T必需为1，也就是说基于单播的多播地址总是临时的

        • plen和prefix两个字段分别为前缀长度和单播前缀的值

          > 标志：| 0 | 0 | 1 | 1 |

|ff|[4b] : [4b] |...|plen|  64bit prefix  |   |   |   |   |   IPv6多播地址
   |<-4:4b范围->|<-       80bit          ->|<- 低32b组ID ->|

    • 特殊的IPv6多播地址：

      • ff01::1和ff02::1是所有节点组，类似IPv4的all-host(224.0.0.1)，但多播是IPv6的组成部分
      • ff01::2、ff02:2、ff05::2是所有路由器组，类似IPv4的224.0.0.2

  1. 多播地址范围

    • IPv6
      • 4bit范围字段
        • 用于指定数据报游走的范围
      • 跳限字段
        • 限制分组被路由器转发
      • 地址范围
        • 1 接口局部(interface-local)：不准接口输出
        • 2 链路局部(link-local)：不可路由器转发
        • 4 管区局部(admin-local)
        • 5 网点局部(site-local)
        • 8 组织机构局部(organization-local)
          • 路由器管理员决定
        • 14 全球或全局的(global)

    • IPv4

      • 没有单独范围字段，TTL字段兼做多播范围
      • 范围
        • 0 接口局部
        • 1 链路局部
        • 2~32 网点局部
        • 33~64 地区局部
        • 65~128 大洲局部
        • 129~255 无范围限制/全球

    • 地址范围

      范围          IPv6    IPv4 [TTL范围      可管理范围]    
接口局部       1              0
链路局部       2              1          224.0.0.0~224.0.0.255  
网点局部       5             <32         237.255.0.0~239.255.255.255
组织机构局部    8                         239.192.0.0.0~239.195.255.255 
全球          14           <=255         224.0.0.1~238.255.255.255

  2. 多播会话

    • 一个多播地址和一个传输层接口确定一个会话(session)
    • 可以用会话区分不同业务

三、局域网上广播和多播比较

  1. 发送多播数据报不用加入多播组，接收端加入
  2. 不具备多播能力的主机忽略多播帧
    • 该帧目的以太网地址不匹配主机的接口地址
    • 该帧的目的以太网地址不是以太网广播地址
    • 该主机的接口没有告知接收任何组地址(没有加入01：00：5e地址)
  3. 不完备过滤
    • 数据链路层->IP层
    • 加入多播组的以太网地址，通常会收到其它以太网组地址为目的地址的帧(因为D类地址前5bit忽略，导致IP->MAC多对一)
    • 数据链路层丢弃以太网多播地址不匹配的数据报
  4. 完备过滤
    • IP层->UDP层
    • 比较多播地址，基于IP层完整32bitD类地址
    • UDP层丢弃不同多播组数据报
  5. 到达应用程序
    • UDP层丢弃相同多播组，不同端口的数据报

四、源特定多播

  1. 广域网上的多播难以部署
    • MRP要求每个多播路由器接收来自所有本地主机的加入多播组和其它请求，并在所有路由器之间交换
    • IPv4没有足够的多播地址可以在广域网上静态分配
  2. 源特定多播SSM结合源地址到组地址
    • 接收进程加入多播组时向多播路由器提供发送进程的源地址
      • 降低多播路由器对每个分组的转发聚散度，因为每个接收进程都必需知道源地址
      • 保留多播地址的包容性，发送进程不需要知道任何接收进程的地址
    • 把多播组标识细化为单播源地址和多播目的地址的组合
      • 发送进程可以挑选任何多播地址，因为源地址和目的地址的组合是唯一的
      • SSM会话由源地址、多播目的地址和端口组合标识
  3. 提供一定的反窃听能力，不同源的数据通道分离，因为源地址不同

五、多播套接字选项

  1. 传统多播套接字选项5个，SSM新增4个

    • 加入/离开的6个只能用于setsocketopt

IP_MUTICAST_TTL        u_char            指定外出多播数据报的TTL
IP_MUTICAST_LOOP    u_char            开启/禁止外出多播数据报的本地回环

IPV6_MUTICAST_IF    u_int            指定外出多播数据报的默认接口            
IPV6_MUTICAST_HOPS    int                指定外出多播数据报的跳限
IPV6_MUTICAST_LOOP    u_int            开启/禁止外出多播数据报的本地回环

    • 组成员相关选项

                                  数据类型                        说明
IP_ADD_MEMBERSHIIP            struct ip_mreq                加入多播组
IP_DROP_MEMBERSHIP            struct ip_mreq                离开多播组
IP_BLOCK_SOURCE                struct ip_mreq_source        在一个已加入组上阻塞某个源
IP_UNBLOCK_SOURCE            struct ip_mreq_source        开通一个早先阻塞的源
IP_ADD_SOURCE_MEMBERSHIP    struct ip_mreq_source        加入一个源特定多播组

IPV6_JOIN_GROUP                struct ipv6_mreq            加入多播组

MCAST_JOIN_GROUP            struct group_req            加入多播组
MCAST_LEAVE_GROUP            struct group_req            离开多播组
MCAST_BLOCK_SOURCE            struct group_source_req        在一个已加入组上阻塞某个源
MCAST_UNBLOCK_SOURCE        struct group_source_req        开通一个早先阻塞的源
MCAST_JOIN_SOURCE_GROUP        struct group_source_req        加入一个源特定多播组
MCAST_LEAVE_SOURCE_GROUP    struct group_source_req        离开一个源特定多播组
      • IPv4多播编程的常见错误时取int类型作为TTL参数

    • 本地接口加入不限源多播组(IP_ADD_MEMBERSHIIP/IPV6_JOIN_GROUP/MCAST_JOIN_GROUP)

      struct ip_mreq {
    struct in_addr imr_multiaddr;            // class D m-addr
    struct in_addr imr_interface;            // addr of local interface
};

struct ipv6_mreq {
    struct in6_addr ipv6mr_multiaddr;        // ipv6 m-addr
    unsigned int     ipv6mr_interface;        // interface index or 0
};

struct group_req {
    unsigned int             gr_interface;    // interface  index or 0
    struct sockaddr_stroage    gr_group;        // ipv4/6 m-addr
};
      • 对IPv4版本，本地接口通过某个单播地址确定，对IPv6和协议无关API，使用接口索引
      • 如果本地接口知道为IPv4通配地址(INADDR_ANY)或IPv6值为0的索引，由内核选择本地接口
      • 一个主机在给定接口上属于某个给定多播组的前提是改主机的一个或多个进程在该接口上属于改组
      • 在一个给定套接字上可以多次加入多播组，但要么多播地址不同，要么在不同接口上加入同一个多播组
      • IPv6显式存在范围字段，如果要不论范围接收所有分组，需要加入：
        • 接口局部(ff01:;101)
        • 链路局部(ff02::101)
        • 网点局部(ff05::101)
        • 组织机构局部(ff08::101)
        • 全球(ff0e::101)
      • 所有这些加入都可在单个套接字上执行，通过设置IPV6_PKTINFO可以让recvmsg返回每个数据报的目的地址
      • IP协议无关的套接字选项(MCAST_JOIN_GROUP)与IPv6版本基本相同，指数用sockaddr_storage结构体代替in6_addr
      • 多数实现对套接字允许加入的次数有限制，通过IP_MAX_MEMBERSHIPS指定
      • 当不指定加入的接口时，Berkeley内核通常在IP路由表中查找给定的多播地址适用的接口
      • IPv6和协议无关的版本通过索引来指定接口，以取代IPv4单播地址指定的方法，意在允许在未指定网络地址的接口或隧道端点执行加入

    • 本地接口离开不限源多播组(IP_DROP_MEMBERSHIIP/IPV6_LEAVE_GROUP/MCAST_LEAVE_GROUP)

      • 如果未指定本地接口，那么抹除首个匹配的多播组成员
      • 如果进程加入多播组后不显式离开，当套接字关闭时，该成员关系自动抹除
      • 单个主机可能多个套接字各自加入相同的多播组，单个套接字上成员关系的抹除不影响改主机继续作为该多播组成员，直到最后一个套接字离开多播组

    • 阻塞来自某个源的多播组(IP_BLOCK_SOURCE/MCAST_BLOCK_SOURCE)

      struct ip_mreq_source {
    struct in_addr imr_multiaddr;        // class D m-addr
    struct in_addr imr_sourceaddr;        // ipv4 source addr
    struct in_addr imr_interface;        // addr of local interface
};

struct group_soruce_req {
    unsigned int             gsr_interface;    // interface  index or 0
    struct sockaddr_storage gsr_group;        // ipv4/6 m-addr
    struct sockaddr_storage gsr_soruce;        // ipv4/6 source addr
};
      • 如果加入同一个多播组的所有套接字都阻塞了相同的源，主机可以通知多播路由器不再需要该分组流通，由此可能影响网络中的多播路由
      • IPv4由单播地址指定接口，协议无关API由索引指定
      • 如果本地接口知道为IPv4通配地址(INADDR_ANY)或IPv6值为0的索引，由内核选择与首个成员匹配的本地接口
      • 阻塞操作是修改已存在的组成员关系，所以要求先加入对应的多播组

    • 开通某个阻塞的源(IP_UNBLOCK_SOURCE/MCAST_UNBLOCK_SOURCE)

      • 如果未指定本地接口，那么开通首个匹配的多播组成员

    • 加入特定源的多播组(IP_ADD_SOURCE_MEMBERSHIIP/MCAST_JOIN_SOURCE_GROUP)

      • 在这个接口上不能加入不限源的该多播组 (同一个套接字能否加入同一个多播组的多个不同源？)
      • 如果本地接口知道为IPv4通配地址(INADDR_ANY)或IPv6值为0的索引，由内核选择本地接口
    • 离开特定源的多播组(IP_LEAVE_SOURCE_MEMBERSHIIP/MCAST_LEAVE_SOURCE_GROUP)
      • 如果未指定本地接口，那么开通首个匹配的特定于源的多播组成员关系
      • 如果加入多播组后不显式离开，当套接字关闭时，该成员关系自动抹除
      • 单个套接字上成员关系的抹除不影响改主机继续作为该多播组成员
    • 指定发送多播数据报的外出接口(IP_MULTICAST_IF/IPV6_MULTICAST_IF)
      • 对IPv4为in_addr指定接口，IPv6由索引指定
      • 如果本地接口知道为IPv4通配地址(INADDR_ANY)或IPv6值为0的索引，之前通过套接字指派的任何接口都被抹除，系统改为每次发送数据报都选择外出接口
      • 区分加入多播组的接口(用于接收)和发送多播数据报的接口(用于发送)
        • 内核查找普通IP路由表选择通往目的路径的接口作为默认外出接口
        • 可能存在一个接口同时用于输入输出
    • 设定IPv4的TTL或IPv6的跳限(IP_MUTICAST_TTL/IPv6_MUTICAST_HOPS)
      • 如果不设定默认为1，从而把多播数据报限制在本子网

    • 开启/禁止多播数据报的本地回环(IP_MUTICAST_LOOP/IPV6_MUTICAST_LOOP)

      • 默认开启，意味着如果一个进程同时属于所发送数据报的目的多播组，就会收到自己发送的所有包
      • 与广播不同的是，本地回环对广播而言无法禁止

    • 上述9个多播套接字选项，前6个影响接收，后3个影响发送

    • 如果发送前没有设置影响发送的套接字选项，数据报的外出接口由内核选择，TTL跳限为1，并且有一个副本自环回来
    • 为接收多播数据报，进程必需加入多播组，并绑定端口到UDP套接字
      • 加入多播组：通知链路层和IP层接收该数据报
      • 绑定端口：通知UDP层将接收数据报到该端口对应的应用程序
    • 有些应用程序将多播地址也绑定到套接字，从而防止IP层把该端口收取的目的地址为其它单播、广播或多播的数据报递送到到套接字。非必要，但通过了内核过滤分组的手段

六、IP分片

  1. 广播报不允许分片
  2. 多播报可以分片

＃ 高级UDP编程
＊＊＊

一、UDP编程的问题

  1. 绑定UDP端口以及通配地址的套接字，能接收这个端口上的单播、多播和广播
  2. 流量控制：收取的数据报大于应用程序输入缓冲区的处理
  3. 超时重传，请求应答，乱序问题
  4. 确定数据报目的地址
    • IP_RECVDSTADDR
    • 绑定所有接口调用select
  5. 某种形式的并发
  6. 每个IPv6数据报的辅助数据指定特定于分组的信息

二、接收标志、目的IP和接口索引

  1. 使用recvmsg接收cmsghdr结构，在msg_control成员中，通过辅助数据返回目的IP和接口索引，在msg_flags中返回标志

    struct cmsghdr {
    socklen_t    cmsg_len;
    int            cmsg_level;
    int            cmsg_type;
};

IPv4    IPPROTO_IP    IP_RECVDSTADDR    随UDP数据报接收的目的地址
                    IP_RECVIF        随UDP数据报接收的接口索引
  2. 返回目的IP如果失败时的处理，可以以0.0.0.0返回，因为全0不可以作为目的地址，只有作为源IP才有效

三、数据报截断

  1. 使用recvmsg接收cmsghdr结构，在msg_flags成员中返回MSG_TRUNC标志
  2. MSG_TRUNC只能从内核返回到进程
  3. 对于超过缓冲区长度，有3种不同的处理方式：
    • 丢弃超过部分，并返回MSG_TRUNC标志
    • 丢弃超过部分，但不通知进程
    • 保留超过部分，在后续的读操作中返回
  4. POSIX采用方式一
  5. 对于不同处理方式，通行的解决办法是：
    • 总是分配比应用程序预期接收的最大数据多一个字节的应用程序buffer，如果收到的长度等于这个buffer大小，就认为是一个过长数据报

四、UDP与TCP

  1. TCP可靠而UDP不可靠
  2. UDP的特性
    • UDP支持广播和多播
    • UDP无连接，没有建立和断开连接的成本
      • UDP只需要两个分组就能交换一个请求和应答
    • 从请求到应答的分组往返次数在延迟超过带宽的情况下变得重要
      • 单个UDP的请求-应答交换的最小事务处理时间为RTT+SPT
        • RTT为round-trip time
        • SPT为server processing time
      • TCP下为2*RTT+SPT
  3. TCP的特性
    • ACK、超时重传、分组乱序与重复检测
    • 基于滑动窗口的流量控制
    • 慢启动和拥塞避免
  4. 总结
    • 广播和多播必须使用UDP
    • 简单的请求-应答可使用UDP，需要完善确认和超时重传
    • 大数据传输使用TCP，关注更好的性能
  5. 例外
    • TFTP使用UDP传输数据
      • 系统引导更容易(UDP 800lines, TCP 4500lines)
      • 只用于局域网，不跨网段
      • 自含顺序确认和超时重传
    • NFS
      • 历史原因
  6. 展望未来
    • 相比TCP，UDP用途在减少，TCP更能充分利用网络带宽
    • 多媒体领域提升UDP的应用，媒体数据通常需要多播

五、给UDP增加可靠性

  1. 使用简单的请求-应答式UDP实现需要做到：
    • 超时重传，以处理丢失的数据报
    • 序列号，以验证应答是否与请求匹配
  2. 请求中分配序列号，应答中回射该序列号，以便于验证应答是否匹配
  3. 发送请求并等待，超时重传直到放弃，这是线性重传定时器
  4. 超时重传的问题在于，RTT的不确定性
    • 从局域网到广域网有很大差异
    • 影响RTT的因素包括
      • 距离
      • 网速
      • 拥塞
    • 需要实测RTT随时间的变化才能确定超时
  5. 需要计算分组的重传超时RTO，并且在重传定时器超时时指数回退
    • 重传定时器满可能的原因：
      • 请求丢失
      • 应答丢失
      • RTO太小
    • 除了序列号外，为每个请求加上时间戳以计算RTT

六、绑定接口地址

  1. 绑定单播地址
    • 设置SO_REUSEADDR选项，为所有IP地址绑定同一个port
  2. 绑定广播地址
    • 允许bind调用返回EADDRINUSE错误，因为如果一个接口上有多个处于同一子网的地址，这些单播地址要对应同一个广播地址
  3. 绑定通配地址
    • 设置SO_REUSEADDR以绑定通配地址

七、并发UDP服务器

  1. 使用TCP时服务器并发的根源在于，每个客户端连接都是唯一的，标识每个连接的是唯一的套接字对
  2. 对UDP服务器，要处理两种不同的情况：
    • 服务器应答以后，客户端不再有请求
      • 这种情况只要收到请求后，新建进程或者线程来发送应答
    • 与客户交换多个数据报
      • 典型的解决办法是让服务器为每个客户端创建一个新的套接字，在其上bind一个临时端口，来进行后续与客户端的交互
      • 要求客户端查看服务器第一个应答的源端口号，把后续的数据发送到该端口
      • TFTP服务器的典型实现

八、IPv6分组信息

  1. IPv6允许为每个外出数据报指定最多5条信息，通过辅助数据来发送，并且拥有对应的套接字粘附选项：
    • 源IPv6地址
    • 外出接口索引
    • 外出跳限
    • 下一跳地址
    • 外出流通类别
  2. IPv6允许为每个接收分组返回4条信息：
    • 目的IPv6
    • 到达接口索引
    • 到达跳限
    • 到达流通类别
  3. 接受指定分组的辅助信息需要开启IPV6_RECVPKTINFO选项
  4. 外出和到达接口
    • IPv6的接口由正整数标识，接口不会为0
    • 如果指定外出接口为0，将由内核选定
    • 由辅助数据指定的接口将覆盖由IPV6_MULTICAST_IF指定的接口
  5. 源和目的地址
    • 源地址通常bind时绑定，指定客户端发送请求的源地址，可以确保服务器应答的源地址等于客户端请求的目的地址 (指客户端选择源地址进行发送，可能书上翻译不好理解)
    • 使用辅助数据指定源地址时，如果指定为通配地址IN6ADDR_ANY：
      • 如果套接字上已经绑定地址，就用其做源地址
      • 如果还没绑定，就由内核选定源地址
  6. 指定和接收跳限
    • 对于单播数据报，外出跳限由IPV6_UNICAST_HOPS套接字选项指定
    • 对于多播数据报，外出跳限由IPV6_MULTICAST_HOPS选项指定
    • 辅助数据指定跳限允许就单次输出指定跳限，以覆盖内核默认值
    • 需要先开启IPV6_RECVHOPLIMIT套接字选项才能接收，cmsg_level成员为IPPROTO_IPV6，cmsg_type成员为IPV6_HOPLIMIT
    • 作为辅助数据返回的值，是来自所接收数据报的真实值，由getsocket返回的IPV6_UNICAST_HOPS套接字选项的值是内核应用于外出套接字上的默认值
    • 指定外出跳限时，把控制信息指定为辅助数据，跳限的正常值在0~255，如果为-1则使用内核通告值
  7. 指定下一跳地址
    • IPV6_NEXTHOP辅助数据为下一跳指定套接字地址结构
    • 下一跳地址必须是发送主机的邻居，如果与目的IPV6地址相等，作用相当于SO_DONTROUTE选项
    • 下一跳地址也可以通过IPV6_NEXTHOP套接字选项为套接字的所有数据报设置，需要超级用户权限
  8. 指定和接收流通类别
    • IPV6_TCLASS辅助数据指定流通类别，需要考题IPV6_RECVTCLASS才能接收
    • 流通类别由DSCP和ECN两个字段构成，必需同时设置
    • 正常值在0~255，如果为-1则使用内核通告值
    • 如果内核需要控制这些值，就会屏蔽或忽略用户的值
    • 可以通过IPV6_TCLASS套接字选项为套接字的所有数据报设置

  9. 数据结构比较

    cmsg_len    cmsg_level    cmsg_type        data
32            IPPRO_IPV6    IPV6_PKTINFO    IPv6地址 接口索引
16            IPPRO_IPV6    IPV6_HOPLIMIT    跳限
40            IPPRO_IPV6    IPV6_NEXTHOP    套接字地址结构
16            IPPRO_IPV6    IPV6_TCLASS        流通类别

九、IPv6路径MTU控制

  1. 最小MTU发送
    • 执行路径MTU发现时，通常按照外出接口与路径MTU中的较小值分配，IPv6定义最小MTU为1280B，必需被所有链路支持
    • 有两种情况可能需要发送最小MTU分组，以避免处理大量的ICMP分片太大消息：
      • 多播
      • 与多个目的地址进行短暂交互
    • IPV6_USE_MIN_MTU套接字选项控制最小MTU
      • 默认值-1表示对多播使用最小MTU，对单播使用路径MTU发现
      • 0表示对所有地址执行路径MTU发现
      • 1表示对所有地址执行最小MTU
    • 这个选项值也可以作为辅助数据发送，cmsg_level成员为IPPROTO_IPV6，cmsg_type成员为IPV6_USE_MIN_MTU

  2. 接收路径MTU变动

    • 开启IPV6_RECVPATHMTU以接收路径MTU变动，作为辅助数据返回。
    • msg_level成员为IPPROTO_IPV6，cmsg_type成员为IPV6_PATHMTU，选项值为ip6_mtuinfo成员

    • 返回数据报的长度可能为0，但含有指示MTU的辅助数据

      #include <netinet/in.h>
struct ip6_mtuinfo {
    struct sockaddr_in6 ip6m_addr;
    uint32_t            ip6m_mtu;
};
  3. 确定当前路径MTU
    • 可以使用IPV6_PATHMTU套接字选项确定某个已连接套接字的当前路径MTU
    • 如果未能获取到路径MTU，就返回外出接口MTU，返回地址值无意义
  4. 避免分片
    • IPV6_DONTFRAG套接字选项用于控制自动分片，默认值0表示允许，1表示关闭
    • 关闭分片后，send返回EMSGSIZE错误表示需要分片，但唯一有效方法是使用IPV6_RECVPATHMTU选项
    • IPV6_DONTFRAG选项也可以作为辅助数据发送，msg_level成员为IPPROTO_IPV6，cmsg_type成员为IPV6_DONTFRAG
