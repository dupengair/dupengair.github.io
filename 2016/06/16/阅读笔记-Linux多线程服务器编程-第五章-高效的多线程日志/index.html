一、概述

  1. 生产环境的日志，应该做到”log everything all the time”
  2. 日志通常要记录关键进程的：
    • 收到的每条内部消息id、关键字段、长度等
    • 收到的外部消息全文
    • 发出的每条消息全文，每条消息有全局唯一的id
    • 关键的内部状态变更
  3. 日志库大体分两部分：前端和后端
    • 前端：客户端接口API，生成日志消息，要求低延迟、开销小，无阻塞
    • 后端：把日志消息写到目的地，要求吞吐量大，资源占用少
  4. 日志库的API风格
    • C/Java的printf(fmt,…)风格，如EdgeOS
    • C++的stream<<风格
  5. stream风格
    • 使用自然，随用随写，不用担心格式化字符串与参数的一致性
    • 类型安全，C++11引入可变参模板后才容易实现，因为C++不允许把非POD对象通过可变参数传入函数
    • 未达到输出日志级别时，打印语句为空操作，基本无运行开销
    • STL的iostream性能低下，可以自己实现，参考muduo

二、功能需求

  1. 需求
    • level：多个消息级别，如TRACE DEBUG INFO WARN ERROR FATAL等
    • appender：多个目的地，如文件、socket、smtp等
    • layout：配置消息格式
    • filter：配置过滤器，控制不同消息的级别和目的地
  2. level可调
    • 必要时应可以临时在线调整输出级别，便于分析查错
  3. 目的地
    • 主要是本地文件，网络容易受影响
    • 写本地文件，通常rolling(滚动)条件有两个：
      • 文件大小
      • 时间
    • 可采取symlink指向最新的日志，每次滚动始终都能找到最新的日志
    • 文件名应包括：
      • 进程名、ID
        • 便于查看进程重启
      • 创建时间(GMT时区)
        • 统一时间便于查看
      • 设备名、MAC或IP等标识
        • 防止集中多个设备日志查看时混淆
      • 统一后缀名
        • 便于脚本过滤
  4. 日志文件压缩与归档应该放到基础设施，便于不同的服务程序共享，也能方便的更好压缩方式或者策略
  5. 写磁盘日志的常见问题
    • 是否应该监控磁盘空间的问题
      • 发生问题时刻的日志是最关键的，如果循环复写日志，容易造成关键日志丢失，应该及时监控到系统异常并且进行干预
    • 程序崩溃，造成日志没有及时flush而丢失
      • 日志不能每条都flush，更不能每条都open/close
      • 解决办法：
        • 定期flush
        • 每条日志都带cookie(哨兵/entry)，记录函数地址，即可在core dump中查找对应的函数信息
  6. 日志格式
    • 格式应该固定
      • 便于工具解析
      • 长时间日志对比，不会应格式变化而造成不便
    • 日志格式要点：
      • 每条一行
        • 容易通过命令行工具awk、sed、grep快速联机查看
      • 时间精确到微妙us，通过gettimeofday(2)获取当前时间
        • 在x86-64linux上，这个函数不是系统调用，不会陷入内核，性能不会有损失
      • 始终使用GMT时区
        • 不用本地转换不同地区的日志时间差
      • 打印日志级别
        • 便于快速查错和发现异常
      • 打印线程id
        • 便于分析多线程程序，发现死锁等
      • 打印文件名和行号
        • 修复bug时准确定位问题
      • 每行日志前几个关键字段宽度固定，以空格分隔，便于脚本解析
      • 避免日志格式出现正则表达式的元字符，如[]，方便使用less(1)进行查看

三、性能需求

  1. 高性能的体现（最低指标）
    • 每秒写上万条日志，没有明显性能损失
    • 能应对一个进程产生大量日志数据的场景，如1G/min
    • 不阻塞正常的执行流程
    • 多线程中，不造成争用contention
  2. 为什么要高性能
    • 日志库应该能瞬时写满磁盘带宽
      • 如带宽110MB/s，每条日志消息平均长度110B，则预期每秒需要能写100万条日志
    • 如果日志能耗尽CPU资源和磁盘带宽的情况下，每秒写入预期的日志量(100W条)，当只需要1秒写入10W条日志时，资源消耗就只有1/10
  3. muduo日志库的优化
    • 时间戳字符串中的日期和时间两部分缓存，一秒内的日志只需重新格式化微秒部分
    • 日志消息前4个字段定长，避免运行期求字符串长度(strlen)。对定长的内存复制，直接memcpy，编译器会在编译期inline展开为高效代码
    • 线程id为预先格式化字符串，输出日志消息只需要拷贝字符
    • 每行日志消息的源文件名采用编译期计算获取的basename，通过gcc内置函数避免运行strchr(3)开销

四、多线程异步日志

  1. 线程安全
    • 采用全局mutex或者每个线程单独一个日志文件的办法，性能低下
      • mutex造成所有线程抢锁
      • 线程单独日志文件造成业务阻塞在写磁盘，且不一定提速(内核操作排队)
    • 多线程最好只写一个日志文件，便于分析
      • 一个背景线程收集日志消息进行写入
      • 业务线程只管发送消息，异步日志
  2. muduo库的双缓冲技术
    • 两个buffer：
      • 前端收集客户端日志消息写入bufferA，后端将bufferB的数据写入文件
      • bufferA写满后交换AB，如此往复
      • 每3秒执行一次交换buffer，不论bufferA是否写满
    • 好处：
      • 写日志消息不用等磁盘IO，先缓冲到buffer
      • 可以把多条日志拼成一个大buffer一次性写入，不用每条日志都写磁盘

五、其它方案

  1. 使用BlockingQueue传递消息
    • 每条消息是一个string，导致每条日志消息都要分配内存
    • 前端线程分配内存(写日志)，后端线程释放，需要对多线程优化
  2. 用bucket减小临界区
    • muduo的异步log实现只用了一个mutex，线程多时可能造成lock contention(锁争用)
    • 可以用多个bucket，前根据线程id，在写日志时hash到不同的bucket中，实现复杂

  3. muduo日志只允许指定文件名，不能指定路径，默认当前路径，以简化实现

    • 类似内核core dump，为了防止core dump冲掉旧的，可以设置名称格式：

      sysctl kernel.core_pattern
      • 见 man 5 core
